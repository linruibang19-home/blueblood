## 二、8 大技术亮点(面试讲解素材)

### 亮点 1:任务接单的并发安全(防超卖 + 防重复)⭐核心

**是什么**:多用户同时抢一个名额有限的任务,不能超卖;同一用户不能重复接。

**为什么难**:朴素的"查名额→判断>0→扣减→插入订单"是**读-改-写**,两个并发请求都读到 slots_left=1,各自扣减,导致超卖或重复订单。

**怎么做**:
- **超卖**:改用原子更新 `UPDATE task SET slots_left = slots_left - 1 WHERE id=? AND slots_left > 0 AND deleted_at IS NULL`,靠**影响行数=0** 判断名额满(MyBatis-Plus `@Update` 自定义方法)
- **重复接单**:DB 层 `UNIQUE KEY uk_order_pair(task_id, user_id)` 兜底,代码捕获 `DuplicateKeyException` 转友好提示;前置 `hasOrder` 检查做快速失败

**怎么讲**:"我用的是乐观思路 —— 不加锁,把扣减做成原子 SQL 让数据库用行锁保证 slots_left 不会被两个事务同时读到旧值;重复接单则靠唯一索引在 DB 层兜底,而不是只依赖应用层查询,这样即使应用层检查有并发漏判,DB 也能拦住。"

### 亮点 2:里程碑分阶段结算 + 幂等入账 ⭐核心

**是什么**:任务有多个里程碑(各带金额),雇主审核通过一个,就立刻把该里程碑的钱打进接单者钱包;不是整体验收后一次性结。

**为什么难**:审核可能重试/并发/重放,绝不能重复入账(资损)。

**怎么做**:
- `WalletService.credit(userId, amount, bizType='task', bizId=里程碑id)`,入账前 `SELECT COUNT(*) WHERE biz_type='task' AND biz_id=? AND type='income'`,已存在则跳过(**幂等**)
- 余额变更在 `@Transactional` 内,`balance += amount`、`total_earned += amount`,写 `wallet_record`
- `wallet_record` 加 `(biz_type, biz_id)` 索引加速查重
- 全部里程碑 APPROVED → 订单 `settled`(`recomputeOrderProgress` 判定)

**怎么讲**:"结算幂等我用的是'业务唯一键去重' —— 每笔入账带 biz_type+biz_id,入账前查重,保证同一里程碑无论被审核多少次只入账一次。比'加分布式锁'轻量,且天然适配并发重试。"

### 亮点 3:JWT 无状态认证 + RBAC 双层鉴权

**是什么**:登录签发 JWT,后续请求 Filter 解析验签;权限分两层。

**怎么做**:
- **认证层**:`JwtAuthenticationFilter`(OncePerRequestFilter)解析 token → 构建 `Authentication` 存入 `SecurityContextHolder`
- **授权层 1(角色)**:后台接口 `@PreAuthorize("hasRole('ADMIN')")`,RBAC 表 `sys_role/sys_user_role`
- **授权层 2(业务)**:service 内校验 —— 发布限 `user_type=enterprise`、订单只让 owner/employer/admin 看。这层不用注解(因为按数据归属判断,非角色)

**怎么讲**:"我用两层鉴权:粗粒度用 Spring Security 的方法注解拦角色(ADMIN),细粒度用业务级判断拦数据归属(这个订单是不是你的/你是不是这个任务的雇主)。注解搞不定'这个资源归不归你'这种判断,必须在 service 里做。"

### 亮点 4:软删除的工程实践(规避 MyBatis-Plus 全局逻辑删除坑)

**是什么**:全部表用 `deleted_at` 软删,**没有**启用 MyBatis-Plus 的全局逻辑删除,而是手动 `.isNull(Entity::getDeletedAt)`。

**为什么**:MyBatis-Plus 全局逻辑删除默认用 `0/1` 标志位;若要用 `deleted_at` 这种 datetime 字段做逻辑删除,需额外配置且容易踩坑(空值/类型/查询拼接)。手动 `.isNull` 显式可控,每个查询自己负责。

**怎么做**:`BaseEntity.deletedAt`(`@TableField(select=false)` 默认不查出来),所有查询显式 `.isNull(...)`。

**怎么讲**:"我评估过 MyBatis-Plus 的全局逻辑删除,但它对 datetime 类型的 deleted_at 支持不友好,容易在 update/查询里出隐性 bug。我选择手动控制 —— 虽然每个查询要多写一行,但行为完全可预期,排查问题也直接。"

### 亮点 5:统一工程规范(响应/异常/分页/填充)

**是什么**:全项目一套规范,任何接口行为一致。

**怎么做**:
- **响应**:`Result<T>{code,message,data,success}`,前端拦截器只看 code
- **异常**:`BusinessException(ResultCode,msg)` + `@RestControllerAdvice` 统一转 JSON(业务异常 HTTP 200+错误码,参数异常 400,认证 401,权限 403)
- **分页**:`PageQuery` + `PageResult.of(IPage)`,统一 `{list,total,page,pageSize}`
- **填充**:`MyMetaObjectHandler` 自动填 createdAt/updatedAt
- **校验**:Bean Validation + 全局异常转 400

### 亮点 6:多端共用一套后端 API

**是什么**:H5 / 后台 / 小程序三端,共用一套 `/api/**`,靠 JWT + 角色区分能力,不各自建后端。

**价值**:业务逻辑单点维护,三端一致;新增端只需写前端。

### 亮点 7:前后台业务域解耦设计

**是什么**:用"业务域"做前后台映射桥梁(社区/教育/机会/财务/账户/运营),前台 4 tab ↔ 后台 6 组清晰对应,每个功能唯一归属。

**怎么做**:见 [02-业务架构](02-业务架构.md)对应表。把"黑客松/岗位"从成长移到任务(同为变现机会域),后台合并"任务管理+活动管理"为"机会管理"。

### 亮点 8:Docker 全栈编排

**是什么**:`docker-compose.full.yml` 编排 5 服务,一键 `up -d --build` 起全栈。

**细节**:
- 端口错开本机原生服务(mysql→3307, redis→6380, api→8090)
- mysql 挂载 `init.sql` 自动建库建表
- api 用多阶段/本地 jar 构建;前端 nginx 托管 dist
- `depends_on: condition: service_healthy` 保证启动顺序
