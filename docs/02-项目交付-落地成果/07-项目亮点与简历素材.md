## 二、技术亮点(分三大类,共 14 项)

### 🔥 A. 并发与稳定性亮点(本项目最大亮点,面试主打)

#### 亮点 1:Redis Lua + DB 双层秒杀架构 ⭐⭐⭐⭐⭐

**是什么**:任务抢单(名额限量)本质是秒杀。用 Redis 内存层挡瞬时高并发,DB 兜底保证正确性。

**为什么难**:朴素"查→判→扣"会超卖;单 DB 扛不住万级 QPS。

**怎么做**(双层):
- **L1 MySQL 原子 SQL**(正确性):`UPDATE task SET slots_left=slots_left-1 WHERE id=? AND slots_left>0`,行锁 + 影响行数判满
- **L2 Redis Lua 预扣**(性能):Lua 脚本 `GET判>0 → DECR → 返回`,Redis 单线程 + Lua 不可打断 = 原子;纯内存扛高并发
- **双写流程**:Redis 预扣成功 → DB 兜底扣 + 写订单 → DB 失败则回补 Redis

**怎么讲**:"Redis 挡流量、DB 保正确。Lua 保证'判断+扣减'原子,DB 原子 SQL 是最后防线,即使 Redis/DB 短暂不一致也不会超卖。这是用最终一致换高并发的标准秒杀做法。"(完整逐字稿见 [11](11-面试深讲脚本-秒杀与Redis一致性.md))

#### 亮点 2:Redis 与 DB 双写一致性(回补机制)⭐⭐⭐⭐⭐

**是什么**:Redis 预扣后若 DB 失败,要保证两边最终一致。

**为什么难**:`@Transactional` 只回滚 DB,**回滚不了 Redis**——Redis 扣的名额会"漏"掉,可用名额越来越少。

**怎么做**:Redis 预扣后用 `try-catch` 包住整个 DB 操作,catch 到任何异常先 `rollback` 回补 Redis 再抛出。

**怎么讲**:"关键坑是 @Transactional 不管 Redis。我用 try-catch 兜底回补,保证 Redis 与 DB 最终一致。接受短暂不一致(少卖),换高并发——超卖由 DB 原子 SQL 兜底,不会发生。"

#### 亮点 3:Redis + Lua 分布式限流 ⭐⭐⭐⭐

**是什么**:`@RateLimit` 注解 + AOP 切面,验证码 60s/次、登录 60s/5 次,防接口刷量。

**怎么做**:Lua 脚本 `INCR + 首次 EXPIRE + 超限判定`原子执行(固定窗口);key 支持 SpEL(`#a0`/`#a0.username`)按业务维度限流,空则按 IP。

**怎么讲**:"限流用 Redis Lua 做 INCR+EXPIRE 原子计数,注解 + AOP 声明式接入,SpEL 灵活指定限流维度(按手机号/账号/IP)。比 Guava 单机限流多了分布式能力。"

#### 亮点 4:任务大厅缓存 + 防击穿/穿透/雪崩 ⭐⭐⭐⭐

**是什么**:任务大厅列表走 Redis 缓存,三类经典缓存问题全防护。

**怎么做**(`TaskListCacheService`):
- **防击穿**:热点 key 过期时 `tryLock` 互斥,只让一个线程重建,其余短暂等待读缓存
- **防穿透**:空结果也缓存(`NULL_MARK`,短 TTL),避免恶意查询反复打 DB
- **防雪崩**:正常结果随机 TTL(30~60s),避免同时失效
- 写操作(发布/编辑/下架)`evictAll` 清缓存

**怎么讲**:"缓存三大问题我各用对应手段:互斥锁防击穿、空值标记防穿透、随机 TTL 防雪崩。这是缓存设计的标配。"

#### 亮点 5:幂等 Token 防重提交 ⭐⭐⭐

**是什么**:发布任务(无天然唯一约束)防前端重复点击/网络重放。

**怎么做**(`IdempotentService`):前端进页取一次性 token,提交带 `X-Idempotent-Token` 头;后端 Redis `DEL` 原子消费——返回 1=首次、0=重复。

**怎么讲**:"幂等有几种手段:唯一索引(重复接单)、业务键去重(结算)、状态机(审核)、Token(发布)。我按场景选最轻量的,不滥用锁。"

#### 亮点 6:原子更新防丢失更新(钱包/计数器)⭐⭐⭐⭐

**是什么**:钱包余额、浏览数这类"读-改-写"在高并发下会**丢失更新**(两事务都读到 100,各 +50,写回 150 而非 200)。

**怎么做**:全部改原子 UPDATE——`addBalance`(balance=balance+?)、`deductBalance`(WHERE balance>=?)、`incrementViewCount`(view_count=view_count+1)。行锁保证无丢失。

**怎么讲**:"这是个容易被忽略的隐患——read-modify-write 看着对,并发下会丢更新。我把所有余额/计数变更改成原子 UPDATE,行锁兜底。这个点能讲出来,说明真踩过并发坑。"

#### 亮点 7:结算分阶段入账 + 幂等 ⭐⭐⭐⭐

**是什么**:多里程碑任务,雇主审一个结一个(不是整体验收后一次结);审核可重试,绝不重复入账。

**怎么做**:每里程碑 APPROVED 即 `credit` 入账;`(biz_type=task, biz_id=里程碑id)` 业务键查重,已存在跳过;全部 APPROVED → 订单 settled。

**怎么讲**:"结算幂等用业务唯一键去重,比分布式锁轻量,天然适配并发重试。分阶段结算是产品 + 技术的结合点。"

### 💼 B. 核心业务亮点

#### 亮点 8:JWT 无状态认证 + RBAC 双层鉴权

**怎么做**:
- **认证**:`JwtAuthenticationFilter` 解析 token → 存 SecurityContext
- **授权 1(角色)**:`@PreAuthorize("hasRole('ADMIN')")` + RBAC 表
- **授权 2(业务)**:service 内校验数据归属(订单 owner/任务 employer)+ user_type(发布限企业)

**怎么讲**:"两层鉴权:注解管功能权限(角色),service 管数据权限(这个资源归不归你)。注解搞不定数据归属判断。"

#### 亮点 9:多端共用一套后端 API

H5/后台/小程序三端共用 `/api/**`,业务逻辑单点维护,新增端只写前端。

#### 亮点 10:前后台业务域解耦设计

用"业务域"做映射桥梁(社区/教育/机会/财务/账户/运营),前台 4 tab ↔ 后台 6 组清晰对应,每功能唯一归属。详见 [02-业务架构](02-业务架构.md)。

## 四、高并发岗位专项简历段(针对性投递)

> 投"要求高并发/中间件"的岗位时,用这段替换/补充上面的项目段,聚焦并发:

> **核心:高并发任务抢单系统的设计与实现**
> - 针对"名额限量抢单"的秒杀场景,设计 **Redis Lua 预扣库存 + DB 原子 SQL 兜底**的双层架构:Redis 单线程 + Lua 脚本保证"判断+扣减"原子,内存层扛瞬时高并发;DB `WHERE slots_left>0` 原子扣作为最后防线防超卖
> - 解决 **Redis 与 DB 双写一致性**:`@Transactional` 不回滚 Redis,用 try-catch + 回补机制保证预扣失败时 Redis 名额归还,最终一致(接受短暂不一致换高并发,超卖由 DB 兜底)
> - 基于 **Redis + Lua** 实现分布式限流(`@RateLimit` + AOP,INCR+EXPIRE 原子),防验证码/登录接口刷量
> - 任务大厅列表 **Redis 缓存 + 三防**(互斥锁防击穿/空值标记防穿透/随机TTL防雪崩)
> - **幂等 Token**(Redis 原子 DEL 消费)防发布重复提交;钱包余额/计数器全部**原子 UPDATE** 防丢失更新
> - 全部并发能力**端到端实测验证通过**
