## 一、业务框架

### 1.1 平台定位
**蓝血菁英平台** —— 面向 AI 人才的「成长 + 变现」一体化平台。学习课程、接任务、参赛/求职、赚收益,形成"学习 → 实战 → 结算"闭环。本项目是其 Java 重构版(原 Laravel/PHP)。

### 1.2 六大业务域 + 用户角色
平台按 **6 个业务域**组织,前台 4 tab 与后台 6 组通过业务域一一映射:

| 业务域 | 前台 | 后台 |
|--------|------|------|
| 社区 | 发现 | 社区管理 |
| 教育 | 成长(课程) | 教育管理 |
| 机会 | 任务(任务+黑客松+岗位) | 机会管理 |
| 财务 | 我的·钱包 | 财务管理 |
| 账户 | 我的·资料 | 平台运营·用户 |
| 运营(后台专属) | — | 工作台 + 平台运营 |

**三类角色**:
- 个人用户(`personal`):学习、接单、社区、提现、可发任务
- 企业用户(`enterprise`,认证后):个人全部 + 发布岗位/黑客松/任务 + 雇主验收
- 管理员(`ADMIN`):后台全部管理 + 兜底审核

### 1.3 核心业务闭环(任务撮合 + 分阶段结算)
```
发布任务(配多里程碑,各带金额) → 大厅展示
  → 用户接单(生成订单 + 复制里程碑实例,原子扣名额)
  → 接单者提交里程碑(成果 + 附件)
  → 雇主(发布者)审核:APPROVED 即按金额入账接单者钱包 / REJECTED 可重提
  → 全部里程碑通过 → 订单 settled
  → 提现申请 → 管理员审核 → 打款 / 驳回退款
```

这条闭环是平台商业化核心,也是技术难点最集中的地方(并发、幂等、状态机、资损防护)。

## 二、技术框架

### 2.1 整体分层
```
前端(H5/后台/小程序) ──Bearer JWT──▶ Spring Boot API(统一后端)
                                          ├── common(响应/异常/分页/基类)
                                          ├── security(JWT 全家桶)
                                          ├── config(Security/MP/Redis/CORS)
                                          └── modules(12 业务模块,各 controller/service/mapper/entity/dto)
                                                  ├── MySQL 8(主数据,44+表)
                                                  └── Redis 7(验证码/缓存预留)
```

### 2.2 请求链路(一次鉴权请求)
```
HTTP → CorsFilter → JwtAuthenticationFilter(解析验签→存 SecurityContext)
     → SecurityFilterChain(路径鉴权) → @PreAuthorize(角色)
     → Controller(@Valid) → Service(业务/事务/校验) → Mapper(MySQL,自动软删过滤)
     → Result<T> 统一响应 / 异常走 GlobalExceptionHandler
```

### 2.3 关键架构决策
- **单后端多前端**:一套 `/api/**` 服务三端,业务逻辑单点维护
- **无状态认证**:JWT,服务端不存 session,易扩容
- **双层鉴权**:角色注解(功能权限)+ service 内判断(数据权限/用户类型)
- **统一规范**:`Result<T>` / `BusinessException` + 全局异常 / `PageResult` / 软删手动 `.isNull` / 自动填充审计字段

## 三、技术选型考量(为什么选 + 解决了什么)

> 选型不是"哪个新用哪个",而是"在项目约束下,哪个成本/收益最优"。

### 3.1 Spring Boot 3 + Java 17
- **为什么**:当前 LTS 主流,Jakarta 命名空间,生态成熟,招聘市场标配
- **解决了**:统一的工程脚手架 + 自动装配 + Actuator 可观测,团队接手成本低

### 3.2 MyBatis-Plus(而非 JPA/Hibernate)
- **为什么**:MP 对 SQL 可控(复杂查询/软删/动态条件好写),自带分页/填充;JPA 自动 SQL 在复杂业务难调优
- **解决了**:本项目大量"按业务域拼条件"的查询(任务大厅筛选、后台多条件分页),Lambda 查询直观;软删手动控制避免隐性 bug

### 3.3 JWT(而非 Session)
- **为什么**:无状态、多端友好(三端都带 token)、前后端分离跨域简单
- **解决了**:多端共用一套认证;水平扩容无需共享 session 存储
- **权衡**:JWT 难主动失效,预留 Redis 黑名单扩展点

### 3.4 Redis(轻量介入)
- **为什么**:验证码需要 TTL 自动过期 + 高频读写,MySQL 不合适
- **解决了**:验证码存取(`verify:{type}:{target}`,TTL 300s)
- **刻意克制**:没滥用缓存(QPS 不大时引入缓存一致性是负担),只在"必须用"处用

### 3.5 软删除手动 `.isNull`(而非 MP 全局逻辑删除)
- **为什么**:MP 全局逻辑删除对 datetime 类型 `deleted_at` 支持差,易踩坑
- **解决了**:行为完全可控可预期,排查直接;代价是每查询多一行,换取确定性

### 3.6 Docker Compose 全栈编排
- **为什么**:多服务(mysql/redis/api/两前端)一键起,环境一致
- **解决了**:端口冲突(错开 3307/6380/8090)、数据自动初始化(mysql 挂 init.sql)、启动顺序(depends_on healthy)

## 四、核心亮点(技术内核)

| 亮点 | 内核 |
|------|------|
| **接单并发安全** | 原子 SQL + 唯一索引,不引入分布式锁 |

| **前后台业务域解耦** | 4 tab ↔ 6 组清晰映射,每功能唯一归属 |

## 五、遇到的困难与解决(实战踩坑)

| 困难 | 现象 | 根因 | 解决 |
|------|------|------|------|
| **接单超卖** | 并发下名额变负 | 读-改-写非原子 | 原子 SQL `WHERE slots_left>0` |
| **重复接单** | 同用户并发点接两次 | 应用层查询有竞态 | DB 唯一索引兜底 + 捕获 DuplicateKey |
| **结算重复入账** | 审核重试多打钱 | 无幂等 | `(biz_type,biz_id)` 业务键去重 |
| **提现驳回资损** | 驳回不退款,用户钱没了 | REJECTED 分支漏退 balance | 补 `balance += amount` |
| **进度卡死** | 订单永远进不了结算 | REJECTED 计入进度分母 | 分母排除 REJECTED |
| **退出登录无效** | 退出后仍登录态 | 未清 token + dev 自动登录回灌 | 清 token/store + 移除自动登录 |
| **企业注册 401** | 注册后提交认证失败 | 注册后无登录态 | 自动登录拿 token 再提交 |
| **软删漏过滤** | 删了的数据还出来 | 部分查询忘 `.isNull` | 统一规范 + CR 把关 |
| **部署坑** | 改 Java 不生效/8080 被占/中文乱码 | Dockerfile COPY jar / 端口 / GBK | mvn package 后 build / 错开端口 / ASCII |

**通用反思**:涉及金额/状态机的逻辑,必须有"反向路径"测试(驳回、失败、并发);应用层校验不够,DB 约束兜底才可靠。

## 六、并发问题设计方案(★ 重点)

### 6.1 项目里的并发场景全景

| 场景 | 并发来源 | 风险 |
|------|---------|------|
| **接单抢单** | 多用户同时抢最后名额 | 超卖(slots_left 变负) |
| **重复接单** | 同一用户并发点击/重试 | 生成重复订单 |
| **结算重复入账** | 雇主重试/网络重放 | 钱打多次(资损) |
| **钱包余额竞争** | 同一用户多笔入账同时 | 丢失更新(balance 错算) |
| **验证码并发** | 同手机号频繁请求 | 状态错乱/刷接口 |

下面分别讲 MySQL 和 Redis 怎么解。

### 6.2 MySQL 层并发方案

#### 方案 A:原子 UPDATE(接单防超卖)⭐
```sql
UPDATE task SET slots_left = slots_left - 1
WHERE id = #{taskId} AND slots_left > 0 AND deleted_at IS NULL
```
- **原理**:MySQL 的 UPDATE 会对命中行加**排他锁(X 锁/行锁)**,两个并发事务只有一个能持有,另一个阻塞;且 `slots_left > 0` 在加锁后再次判定,保证不会扣成负数
- **判断**:MyBatis 返回**影响行数**,=0 表示名额已被抢完 → 抛"名额已满"
- **为什么不用"查→判→改"**:那是读-改-写,两个事务都读到 slots_left=1 → 都判 >0 → 都扣 → 超卖
- **为什么不用乐观锁(版本号)**:乐观锁要重试,接单场景重试体验差(用户点一次,失败让重试);原子 SQL 一次成功,且 DB 保证
- **为什么不用悲观锁(`SELECT ... FOR UPDATE`)**:能解,但多一次加锁查询,原子 UPDATE 一步到位更轻

#### 方案 B:唯一索引(防重复)⭐
```sql
UNIQUE KEY uk_order_pair (task_id, user_id)   -- task_order 表
UNIQUE KEY uk_wallet_user (user_id)            -- 一人一钱包
UNIQUE KEY uk_ms_milestone (milestone_id)      -- 一里程碑一提交
```
- **原理**:DB 层强约束,即使应用层有并发漏判,重复插入抛 `DuplicateKeyException`
- **代码**:捕获异常转友好提示
- **价值**:把"防重复"从"应用层尽力而为"升级为"DB 强保证"

#### 方案 C:事务保证多表原子(接单/结算/提现)
```java
@Transactional(rollbackFor = Exception.class)
public Map<String, Long> accept(Long taskId) {
    // 插订单 + 复制里程碑 + 原子扣名额,要么全成功要么全回滚
}
```
- **要点**:`@Transactional` 加在 **service 层**(不在 controller/mapper);`rollbackFor=Exception.class` 确保业务异常也回滚(默认只回滚 RuntimeException)

#### 关于事务隔离级别
- MySQL 默认 **RR(可重复读)**,接单的原子 UPDATE 在 RR 下依然安全 —— 因为 **X 锁是当前锁**,不依赖快照
- 没有刻意改成 RC,RR + 行锁 + 唯一索引 已足够;Next-Key Lock 还能防幻读(对本场景多余但无害)

### 6.3 Redis 层并发方案

#### 当前用法:验证码(天然适合 Redis)
```
key: verify:{type}:{target}   value: 6位码   TTL: 300s
```
- **为什么 Redis 而非 MySQL**:验证码是"短时高频 + 自动过期",MySQL 要定时清理过期记录,Redis TTL 天然解决
- **并发防刷**:可加 `INCR verify:limit:{target}` 限频(本项目未做,预留)

#### 关键判断:为什么没用 Redis 分布式锁?
- **评估过**:SET NX EX + Lua 释放 + 看门狗续期是成熟方案
- **但接单场景不需要**:接单的并发问题用 MySQL 原子 UPDATE + 唯一索引就能强保证,且更简单可靠(无锁超时/续期/网络分区问题)
- **Redis 锁适用场景**:跨服务/跨库的临界区、限流、防重。本项目是单库,MySQL 兜底更直接
- **体现的原则**:并发问题优先用"数据本身的能力"(原子 SQL/唯一约束),只有跨数据源时才上分布式锁

#### 如果要引入 Redis 分布式锁,会怎么设计(深度展开)
```
1. 加锁:SET lock_key {threadId+uuid} NX EX 30   (原子:不存在才设 + 过期)
2. 业务执行
3. 释放:Lua 脚本判断 value==自己再加锁的标识 才 DEL  (防误删别人的锁)
4. 续期:看门狗定时(如 1/3 过期时间)续期,防业务超时锁自动释放被别人抢
```
坑点:锁过期 vs 业务未完成的竞态(看门狗解)、主从切换丢锁(Redlock)、网络分区。所以"能用 MySQL 就别上 Redis 锁"。

### 6.4 钱包余额并发安全(诚实复盘 + 改进)

**场景**:同一用户的多笔入账(如多个里程碑同时被雇主审核结算)竞争 `wallet_account.balance`。

**当前实现**(`WalletService.credit`):
```java
@Transactional
public Long credit(userId, amount, ...) {
    WalletAccount acc = getOrCreateAccount(userId);   // 读
    acc.setBalance(acc.getBalance() + amount);        // 改
    walletAccountMapper.updateById(acc);              // 写
}
```
这是**读-改-写**,在极高并发下有**丢失更新**风险:两个事务都读到 balance=100,各自 +50 写回,结果是 150 而不是 200。

**为什么目前可接受**:
- 结算由雇主**手动审核**触发,同一用户的里程碑结算天然**串行**(雇主一个个审),并发量极低
- 结算幂等已保证"不重复入账",只是"并发同时入账"的理论风险

**正确的并发安全改法**(同接单原子 SQL 思路):
```sql
UPDATE wallet_account SET balance = balance + #{amount}, total_earned = total_earned + #{amount}
WHERE user_id = #{userId}
```
原子加法,行锁保证,无丢失更新。**这是我知道的局限 + 知道怎么改**,面试讲出来是加分项(体现"知其然知其所以然")。

### 6.5 幂等设计(并发的"近亲"问题)
| 手段 | 用在哪 |
|------|--------|
| **唯一索引** | 重复接单、一人一账户、一里程碑一提交 |
| **业务键去重** | 结算入账 `(biz_type=task, biz_id=里程碑id)` |
| **状态机** | 里程碑审核(只有 SUBMITTED 才能 review → APPROVED/REJECTED) |
| **前置查询 + DB 兜底** | 接单 hasOrder 快速失败 + uk_order_pair 兜底 |

**原则**:不滥用锁/Token,按场景选最轻量的;DB 约束 > 业务键去重 > 状态机 > 分布式锁。

### 6.6 并发方案选型矩阵(总结)

| 场景 | 最终方案 | 备选(为什么不用) |
|------|---------|------------------|
| 接单防超卖 | MySQL 原子 UPDATE | 乐观锁(要重试)/ 悲观锁(多一次查询)/ Redis 锁(单库没必要) |
| 防重复接单 | 唯一索引 + 捕获异常 | 应用层查询(有竞态) |
| 结算防重复入账 | 业务键去重(biz_type+biz_id) | 分布式锁(过重) |
| 验证码 | Redis String + TTL | MySQL(要清理过期) |
| 钱包余额(高并发) | **应改**原子 UPDATE `balance=balance+?` | 当前 read-modify-write(低并发够用) |
| 跨服务临界区(未来) | Redis 分布式锁 | 单库不需要 |

## 七、一句话总结

> 这个项目的并发设计哲学是:**优先用数据库自身能力(原子 SQL、唯一索引、事务)解决并发,把强一致性下沉到 MySQL;Redis 只用在它擅长的场景(短时 TTL),不滥用缓存/锁。** 每个并发点都有"为什么这么做"和"还能怎么做"的权衡 —— 这正是后端工程能力的体现。
