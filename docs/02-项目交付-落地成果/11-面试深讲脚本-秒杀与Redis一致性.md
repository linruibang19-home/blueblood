## 一、任务抢单秒杀架构(3 分钟讲透)

### 🎤 逐字稿

> 我这个项目有个**任务抢单**场景:企业发布一个任务,带名额限制,比如 100 个,然后一堆用户去抢。**这个本质就是秒杀**。
>
> 最朴素的做法是:用户点接单,后端先查一下名额还剩多少,大于 0 就扣一个、写订单。但这种"**先查后改**"在高并发下会超卖——两个请求同时查到还剩 1 个,都判断大于 0,各自扣一个,结果名额变成负数。
>
> 所以我做了**两层防护**。
>
> **第一层是 MySQL,解决正确性。** 我把扣名额写成一条原子 UPDATE:`UPDATE task SET slots_left = slots_left - 1 WHERE id = ? AND slots_left > 0`。靠数据库的**行锁**,两个并发事务不会同时扣同一个名额;再通过**影响行数**判断——返回 0 就说明名额已经被抢完了。这一层把超卖彻底解决了,但 MySQL 是单机,真到万级 QPS 它扛不住。
>
> **第二层是 Redis,解决性能。** 接单的时候,我先用一个 **Lua 脚本**在 Redis 里原子地扣名额。为什么要用 Lua?因为 Redis 是单线程,Lua 脚本在执行期间不会被别的命令打断,所以"判断名额大于 0 + 扣减"这两步天然原子。Redis 是纯内存操作,能扛很高并发。**只有 Redis 扣成功了,我才去 DB 写订单、再扣一次 DB 名额兜底。**
>
> 这里有个关键点:如果 DB 那一步失败了——比如名额已经被 DB 层别的请求抢完——我要**把 Redis 扣掉的那个名额还回去**,叫回补。否则名额就漏在 Redis 里了。
>
> **效果就是**:Redis 挡住了绝大部分瞬时流量,DB 只处理"Redis 已经确认有名额"的请求,压力大幅下降;同时 DB 的原子 SQL 是最后一道防线,哪怕 Redis 和 DB 短暂不一致,也绝不会真超卖。这是个典型的**用最终一致换高并发**的取舍。

### 🦴 骨架卡(背这 5 个词就能讲)

`秒杀场景` → `查改写会超卖` → `MySQL原子SQL解决正确性` → `Redis Lua预扣解决性能` → `失败回补保证一致`

### 🎯 追问预案

| 面试官追问 | 怎么答 |
|-----------|--------|
| **Redis 挂了怎么办?** | 降级到 DB 原子 SQL。我做了懒加载:Redis 没 key 时从 DB 读 slots_left 初始化,所以 Redis 挂了正确性不丢,只是性能退化成单 DB |
| **为什么不用消息队列异步落单?** | 这是更彻底的秒杀方案(Redis 预扣 + MQ + 异步落 DB),但项目没引 MQ,我用"Redis 预扣 + 同步 DB 兜底"是同等思路的轻量版。如果 QPS 再上一个量级,我会把 DB 写入挪到 MQ 异步 |
| **Lua 脚本具体怎么写的?** | `GET 判断 > 0` → `DECR` → `返回 1 成功 / 0 满`,三步在 Lua 里原子执行 |
| **Redis 和 DB 名额不一致,谁是对的?** | DB 是唯一真相。Redis 只是缓存层挡流量,以 DB 的原子 SQL 兜底,不一致只会让少数请求被 Redis 误挡(少卖),不会超卖 |
| **为什么不用 `SELECT ... FOR UPDATE` 悲观锁?** | 能解,但多一次加锁查询,原子 UPDATE 一步到位更轻;悲观锁还可能锁竞争/死锁 |

## 二、Redis 与 DB 双写一致性(3 分钟讲透)

### 🎤 逐字稿

> 因为我用 Redis 做了秒杀预扣,就引出一个经典问题:**Redis 和数据库的双写一致性**。
>
> 问题根源在于:**Spring 的 `@Transactional` 只能回滚数据库,回滚不了 Redis。** 接单流程里,我先在 Redis 扣了一个名额,然后去 DB 写订单、扣 DB 名额。如果 DB 这一步抛异常了——最典型的就是**重复接单触发唯一索引冲突**——事务会回滚 DB,订单没建成、DB 名额也没扣,但 **Redis 那个名额已经扣掉了**。不处理的话,这个名额就"漏"在 Redis 里,实际可用名额越来越少。
>
> 我的解决办法很直接:**Redis 预扣之后,用 try-catch 把整个 DB 操作包起来,catch 到任何异常,先 `rollback` Redis 再把异常抛出去。** 这样不管 DB 那一步怎么失败——重复接单、名额满、其他异常——Redis 都会被回补,两边最终对得上。
>
> 这里我要跟面试官强调一个**取舍**:我接受 Redis 和 DB 的**短暂不一致**。比如 Redis 显示剩 2,但 DB 其实还剩 3(还没同步过来),这种不一致只会让少数请求被 Redis 提前判定"满了"而挡掉,**误伤是少卖、不是超卖**。因为 DB 的原子 SQL `WHERE slots_left > 0` 是最后防线,Redis 漏过来的请求到 DB 还会再校验一次。**这是用最终一致换高并发的标准做法,秒杀场景的正确性由 DB 兜底,性能由 Redis 顶。**
>
> 如果面试官追问强一致怎么做,我会说:可以把 Redis 扣 + DB 扣放进一个分布式事务里协调,但成本高、性能反而下来了,**秒杀场景不值得为强一致牺牲性能**——少卖几个名额,远好过系统被打挂。

### 🦴 骨架卡

`@Transactional不管Redis` → `Redis扣了DB失败=漏名额` → `try-catch+rollback回补` → `接受短暂不一致换高并发` → `DB原子SQL兜底,只误伤少卖不超卖`

### 🎯 追问预案

| 面试官追问 | 怎么答 |
|-----------|--------|
| **为什么不做成强一致?** | 强一致要分布式事务(Redis+DB 协调),性能代价大。秒杀的核心是"不超卖",少卖可接受,所以选最终一致 + DB 兜底 |
| **Redis 宕机重启,名额数据丢吗?** | 不丢。DB 是真相,Redis 只是缓存;我做了懒加载,重启后 Redis 无 key 时从 DB slots_left 重新初始化 |
| **如果回补也失败了呢(异常 + Redis 又挂)?** | 极端情况。可以加定时对账任务(Redis 名额 vs DB slots_left)补偿;本项目 QPS 不大,回补失败概率极低,但对账是生产级的兜底手段 |
| **这套和"先写 DB 再删缓存"有啥区别?** | 那是缓存模式的读写策略(Cache Aside)。我这是**库存扣减**场景,Redis 不是普通缓存而是"预扣的库存",所以用"预扣+回补"而不是"写DB删缓存" |
| **Lua 脚本执行中途 Redis 宕机?** | Lua 是原子的,要么全执行要么不执行,不会扣一半。宕机的话这条请求失败重试即可,DB 没动 |

## 三、通用:技术方案讲解"五步法"

任何技术点都能用这个骨架讲,不会乱:

```
1. 场景  —— 我的项目里有个 XX 场景(把抽象问题落到你的业务)
2. 问题  —— 朴素做法在 XX 条件下会出 YY 问题(点出痛点)
3. 方案  —— 我分 N 层解决(分层体现深度,别只说一层)
4. 关键  —— 核心是 XX 技术,因为它 YY(讲清为什么用它)
5. 效果/取舍 —— 带来 ZZ 提升;这里我接受了 AA 换 BB(体现权衡意识)
```

> 第 1 步"落到你的业务"是关键——面试官最烦听八股,你把"超卖"说成"我项目里任务抢单",立刻就具体了。

## 四、练习建议

1. **先背骨架卡**(5 个词),对着骨架自己讲一遍,卡壳的地方就是没吃透的
2. **录音听一遍**,标记"嗯啊"和逻辑断层,练到能连贯
3. **模拟追问**:让朋友/自己按"追问预案"提问,练临场反应
4. **控制时间**:每段 3 分钟,超了说明啰嗦,精简到关键点
5. **两个一起练**:面试官常连问"秒杀 → 那一致性怎么保证",两个脚本要能串起来讲
