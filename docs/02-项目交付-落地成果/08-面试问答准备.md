# 08 · 面试问答准备

> 按"技术选型 / 困难难点 / 设计深度 / 效果量化 / 八股结合"组织。每题给项目实例 + 标准答法,可直接用于面试。

## A. 技术选型类("为什么选 X 不选 Y")

### Q1:为什么选 Spring Boot 3 + Java 17?
**答**:Spring Boot 3 是当前 LTS 主流,基于 Jakarta EE 9+(javax→jakarta 命名空间),生态成熟。Java 17 也是 LTS,有 record/sealed/模式匹配等,且是 SB3 的基线要求。选它是为了技术栈现代、长期可维护,也符合招聘市场主流。

### Q2:为什么 MyBatis-Plus 而不是 JPA/Hibernate?
**答**:
- MP 对 SQL 可控(复杂查询写 SQL/Lambda),JPA 的自动 SQL 在复杂业务(多表关联、软删、动态条件)难调优
- MP 自带分页/自动填充/逻辑删除/代码生成,开发效率高
- 国内 Spring Boot 项目 MP 是主流,团队接手成本低
- 本项目有大量"按业务域拼条件"的查询(任务大厅筛选、后台多条件分页),Lambda 查询写起来直观

### Q3:为什么 JWT 而不是 Session?
**答**:
- **无状态**:服务端不存会话,水平扩容无需共享 session 存储
- **多端友好**:H5/后台/小程序三端都只需带 token,统一
- **前后端分离**:跨域场景下 cookie+session 麻烦,JWT 放 Header 简单
- **权衡**:JWT 没法主动失效(登出仅客户端丢 token),我预留了 Redis 黑名单的扩展点(`AuthService.logout` 注释 TODO)。要严格失效可加黑名单或用 refreshToken 方案。

### Q4:为什么 Redis 只用来存验证码,没做缓存?
**答**:当前业务读写频率不高(QPS 不大),MySQL + 索引够用,避免引入缓存一致性问题。Redis 先用在"必须用"的地方(验证码 TTL、后续 token 黑名单)。如果热点查询(任务大厅列表)QPS 上来,再加缓存层 + 失效策略。**不过度设计**。

### Q5:为什么没启用 MyBatis-Plus 的全局逻辑删除?
**答**:MP 全局逻辑删除对 `0/1` 标志位友好,但本项目用 `deleted_at`(datetime)做软删。强行配置全局逻辑删除容易出现:空值判定、update 时漏拼、类型转换等隐性 bug。我选择**手动 `.isNull(deleted_at)`**,每个查询显式负责,行为完全可预期,排查直接。代价是多写一行,但换来了可控性。

## B. 困难难点类("遇到什么坑,怎么解决的")

### Q6:任务接单高并发下怎么防超卖和重复接单?(高频题)
**难点**:朴素"查→判→扣→插"是读-改-写,并发下两个请求都读到 slots_left=1,各自扣减 → 超卖;重复接单靠应用层查询有并发漏判。

**解决**:
- **超卖**:原子 SQL `UPDATE task SET slots_left=slots_left-1 WHERE id=? AND slots_left>0 AND deleted_at IS NULL`,靠**影响行数=0** 判断满。数据库行锁保证不会两个事务同时读到旧值。
- **重复接单**:DB 唯一索引 `uk_order_pair(task_id,user_id)` 兜底,代码捕获 `DuplicateKeyException` 转友好提示。

**效果**:并发安全从"应用层尽力而为"提升到"DB 层强保证",无需分布式锁。

### Q7:分阶段结算怎么保证不重复入账?
**难点**:雇主审核可能重试/网络重放,绝不能重复打钱。

**解决**:钱包入账 `credit(bizType='task', bizId=里程碑id)` 前查重 —— 同一 `(biz_type,biz_id,type=income)` 已存在则跳过返回。配合 `@Transactional` 保证余额变更与流水写入原子。

**效果**:幂等。任意重试/重放只入账一次,无资损。

### Q8:开发中遇到最难的 bug?
**候选 1 — 提现驳回不退款(资损)**:`AdminWalletService` 审核提现 REJECTED 分支没把冻结的余额退回,用户钱没了。修复:REJECTED 时 `balance += amount`。
**候选 2 — 接单进度卡死**:`recomputeOrderProgress` 把 REJECTED 里程碑算进分母,导致 `approved < total` 永远成立,订单永远进不了结算态。修复:分母排除 REJECTED。
**候选 3 — 邮箱注册 401**:企业注册页注册后没登录态,紧接着提交企业认证申请(需鉴权)→ 401。修复:注册后自动登录拿 token 再提交。

**讲法**:选一个,讲清"现象 → 定位(root cause)→ 修复 → 反思(怎么避免)"。比如资损 bug 反思:涉及金额的状态机要有测试覆盖 REJECTED 分支。

### Q9:软删除踩过什么坑?
**答**:`BaseEntity.deletedAt` 字段,早期有些查询忘了加 `.isNull`,导致软删的数据还出来。后来统一规范:**所有业务查询必须带软删条件**,Code Review 把关。也考虑过 MP 全局逻辑删除,但 datetime 类型支持不好(见 Q5),权衡后手动。

### Q10:多端联调 / 部署的坑?
**答**:
- **端口冲突**:本机 8080 被 Docker 占,api 改用环境变量 `SERVER_PORT=8090`,容器内仍 8080
- **后端改了不生效**:Dockerfile 是 `COPY target/*.jar`,改 Java 必须先 `mvn package` 再 build,不是热加载
- **中文编码**:Windows curl POST 中文被 GBK 打乱,改用 ASCII payload 或文件输入
- **MySQL 启动慢**:init.sql 导入 44 表,api 用 `depends_on: service_healthy` 等它就绪

## C. 设计深度类("如果...你怎么做")

### Q11:JWT 怎么实现登出失效?
**答**:JWT 无状态,签发后服务端无法单方面撤销。当前登出是客户端丢 token(够用于一般场景)。严格失效方案:
1. Redis 黑名单:登出时把 token 存 Redis 到过期,Filter 每次查黑名单
2. 短期 accessToken + 长期 refreshToken,服务端可吊销 refresh
本项目预留了黑名单扩展点(`AuthService.logout` 注释)。

### Q12:幂等有哪几种方案?你用了哪种?
**答**:
- 唯一索引(数据库层)— 本项目重复接单用它
- 业务键去重(查询判断)— 本项目结算入账用它
- token 机制(前端先取 token,提交带 token,用后失效)
- 状态机(某状态才能操作)— 本项目里程碑审核用它(SUBMITTED 才能 review)
- 分布式锁
**选型依据**:看场景。接单靠唯一索引(强一致,简单);结算靠业务键去重(同里程碑多次审核);审核靠状态机(改变状态)。不滥用锁。

### Q13:RBAC 和数据权限怎么分?
**答**:
- **功能权限(RBAC)**:能不能访问这个接口 → `@PreAuthorize(hasRole)` + 角色表
- **数据权限**:能不能操作这条数据 → service 内判断(订单的 userId 是不是你、任务的 employerId 是不是你)
注解做功能权限,业务代码做数据权限,两者结合。

### Q14:事务边界怎么定?
**答**:`@Transactional` 加在 **service 方法**上(不在 controller,controller 不该有事务;也不在 mapper)。涉及多表写入的用事务(接单:订单+里程碑+名额;结算:钱包账户+流水;提现:账户+记录)。`rollbackFor=Exception.class` 确保业务异常也回滚(默认只回运行时异常)。

### Q15:索引怎么设计的?
**答**:
- **唯一索引**:`uk_order_pair(task_id,user_id)` 防重复接单、`uk_wallet_user(user_id)` 一人一账户、`uk_ms_milestone(milestone_id)` 一里程碑一提交 —— 既是业务约束,也是并发兜底
- **查询索引**:status/task_id/user_id 等高频筛选字段
- **联合索引**:`(biz_type,biz_id)` 用于结算幂等查重;顺序按"等值查询字段在前"

## D. 效果量化类("带来了什么提升")

### Q16:并发安全改造前后对比?
**前**:`getTask → 判 slotsLeft>0 → setSlotsLeft(-1) → update`,纯应用层,并发超卖。
**后**:原子 SQL `WHERE slots_left>0`,DB 行锁保证。**效果**:并发正确性从"概率性出错"到"DB 强保证",且无需引入分布式锁,实现简洁。

### Q17:统一工程规范的价值?
**效果**:新接口开发只需关注业务,响应格式/异常处理/分页/校验全自动;前端拦截器一套逻辑处理所有接口;联调成本大幅降低。**这是"工程素养"的体现,面试官看重。**

### Q18:多端共用 API 省了什么?
**效果**:业务逻辑单点,三端一致。新增小程序端时,**后端零改动**,只写前端。避免了"每个端一套后端"的维护地狱。

## E. 八股结合项目(高频知识点 × 项目实例)

| 八股点 | 在本项目体现 |
|--------|-------------|
| HashMap/并发 | — (后端少直接用,可聊 Redis 验证码并发) |
| synchronized/Lock | 本项目**避免**用锁,用原子 SQL + 唯一索引替代(可讲选型) |
| 事务隔离级别 | MySQL 默认 RR,接单原子 SQL 靠行锁;可聊"为什么没用乐观锁版本号" |
| 索引失效 | 软删用 `.isNull(deleted_at)` 能走索引;避免 `!=`/函数/类型转换 |
| Redis 数据结构 | 验证码用 String(TTL);可聊"为什么不用 hash" |
| HTTP 状态码 | 业务异常返 200+错误码(前后端约定),参数 400,认证 401,权限 403 |
| 跨域 CORS | 后端 CorsConfigurationSource 配置,允许凭证 |
| Bean 生命周期 | `@RequiredArgsConstructor` 构造注入 service(Filter/Handler) |
| AOP | 全局异常 `@RestControllerAdvice` 本质是 AOP 拦截 controller |
