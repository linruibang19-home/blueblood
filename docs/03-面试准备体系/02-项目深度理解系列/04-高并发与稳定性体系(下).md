## 一、Redis + Lua 分布式限流

### 1.1 为什么需要限流？

| 场景 | 风险 | 限流策略 |
|------|------|---------|
| 发送验证码 | 恶意刷接口 | 60 秒只能发 1 次 |
| 登录 | 暴力破解 | 60 秒最多试 5 次 |
| 接口调用 | DDoS 攻击 | 按 IP/用户限流 |

### 1.2 限流注解设计

**项目代码**（[RateLimit.java](../../server/blueblood-api/src/main/java/com/blueblood/api/common/ratelimit/RateLimit.java)）：

```java
@Target(ElementType.METHOD)
@Retention(RetentionPolicy.RUNTIME)
public @interface RateLimit {
    /** 时间窗口（秒） */
    int seconds() default 60;
    
    /** 时间窗口内最大请求次数 */
    int count() default 10;
    
    /** 限流 key 的 SpEL 表达式（可选） */
    String key() default "";
    
    /** 超限提示信息 */
    String message() default "请求太频繁，请稍后再试";
}
```

**使用方式**：

```java
// 验证码：60秒只能发1次
@RateLimit(seconds = 60, count = 1, key = "#phone", message = "发送太频繁，请60秒后再试")
public Result<?> sendCode(@RequestParam String phone) { ... }

// 登录：60秒最多试5次
@RateLimit(seconds = 60, count = 5, key = "#req.phone", message = "登录太频繁，请60秒后再试")
public Result<LoginResponse> login(@RequestBody LoginRequest req) { ... }
```

### 1.3 Lua 脚本实现

**项目代码**（[RateLimitAspect.java](../../server/blueblood-api/src/main/java/com/blueblood/api/common/ratelimit/RateLimitAspect.java)）：

```java
private static final DefaultRedisScript<Long> SCRIPT;
static {
    SCRIPT = new DefaultRedisScript<>();
    SCRIPT.setScriptText(
            "local c = redis.call('INCR', KEYS[1]) " +        // 1. 计数器 +1
            "if tonumber(c) == 1 then " +                    // 2. 首次访问
            "    redis.call('EXPIRE', KEYS[1], ARGV[1]) " + //    设置过期时间
            "end " +
            "if tonumber(c) > tonumber(ARGV[2]) then " +    // 3. 超过次数？
            "    return 0 " +                                //    返回0（限流）
            "else " +
            "    return 1 " +                               //    返回1（放行）
            "end");
    SCRIPT.setResultType(Long.class);
}
```

**脚本执行流程**：

```
┌─────────────────────────────────────────────────────────┐
│                 固定窗口限流执行流程                     │
├─────────────────────────────────────────────────────────┤
│                                                          │
│  请求1：                                                 │
│  INCR key → 1                                          │
│  c == 1 → EXPIRE key 60                                │
│  c(1) <= count(5) → return 1 ✓                         │
│                                                          │
│  请求2：                                                 │
│  INCR key → 2                                          │
│  c != 1 → 不设置过期                                    │
│  c(2) <= count(5) → return 1 ✓                         │
│                                                          │
│  ...                                                    │
│                                                          │
│  请求6：                                                 │
│  INCR key → 6                                          │
│  c(6) > count(5) → return 0 ✗（限流）                   │
│                                                          │
└─────────────────────────────────────────────────────────┘
```

### 1.4 AOP 切面实现

```java
@Aspect
@Component
@RequiredArgsConstructor
public class RateLimitAspect {
    
    @Around("@annotation(rateLimit)")
    public Object around(ProceedingJoinPoint pjp, RateLimit rateLimit) throws Throwable {
        // 1. 解析限流 key
        String key = "rate:" + resolveKey(rateLimit, pjp);
        
        // 2. 执行 Lua 限流脚本
        Long allowed = redis.execute(SCRIPT, Collections.singletonList(key),
                String.valueOf(rateLimit.seconds()), String.valueOf(rateLimit.count()));
        
        // 3. 超限则抛异常
        if (allowed == null || allowed == 0L) {
            throw new BusinessException(ResultCode.OPERATION_FAILED, rateLimit.message());
        }
        
        // 4. 放行
        return pjp.proceed();
    }
    
    /** 解析限流 key：支持 SpEL 表达式 */
    private String resolveKey(RateLimit rl, ProceedingJoinPoint pjp) {
        if (rl.key() == null || rl.key().isEmpty()) {
            return clientIp();  // 空则按 IP 限流
        }
        // 解析 SpEL：#phone 或 #a0
        // ...
    }
}
```

### 1.5 限流算法对比

| 算法 | 原理 | 优点 | 缺点 |
|------|------|------|------|
| **固定窗口**（项目用） | 固定时间窗口内计数 | 简单 | 临界突变 |
| 滑动窗口 | 滑动时间窗口，更精确 | 精确 | 实现复杂 |
| 令牌桶 | 以固定速率放入令牌 | 允许突发 | 实现复杂 |
| 漏桶 | 以固定速率消费 | 平滑限流 | 不允许突发 |

**临界突变问题**：

```
固定窗口（有问题）：
窗口1（0-60s）：60次请求 → 窗口重置
窗口2（60-120s）：60次请求

问题：第60秒和第61秒各有60次请求，共120次，窗口内最多允许60次

滑动窗口（无问题）：
请求1（第60s）：统计第0-60s = 60次
请求2（第61s）：统计第1-61s = 60次
请求3（第62s）：统计第2-62s = 60次
（滚动计算，平滑）
```

## 二、缓存三防

### 2.1 概述

| 问题 | 原因 | 后果 | 解决方案 |
|------|------|------|---------|
| **穿透** | 查询不存在的数据 | 每次都打 DB | 空值缓存 |
| **击穿** | 热点 key 过期瞬间 | 大量请求打 DB | 互斥锁 |
| **雪崩** | 大量 key 同时过期 | 缓存雪崩，DB 被打垮 | 随机 TTL |

### 2.2 防穿透：空值标记

**问题**：恶意查询一个不存在的 key，每次都穿透到 DB。

**解决方案**：

```java
public PageResult<TaskListItemVO> get(String cacheKey) {
    Object v = redis.opsForValue().get(KEY_PREFIX + cacheKey);
    
    if (v == null) {
        return null;  // 未命中，继续查 DB
    }
    
    if (NULL_MARK.equals(v)) {
        return emptyPage();  // 空值缓存，直接返回空结果
    }
    
    return (PageResult<TaskListItemVO>) v;
}

public void set(String cacheKey, PageResult<TaskListItemVO> data, boolean empty) {
    if (empty) {
        // 空结果：写 NULL_MARK，短 TTL（10秒）
        redis.opsForValue().set(KEY_PREFIX + cacheKey, NULL_MARK, Duration.ofSeconds(10));
    } else {
        // 正常结果：正常缓存
        redis.opsForValue().set(KEY_PREFIX + cacheKey, data, Duration.ofSeconds(30));
    }
}
```

**流程**：

```
请求1：查 key=A
→ Redis GET A = null
→ DB 查 = 空
→ 写 Redis A = "__NULL__"，TTL=10s
→ 返回空

请求2：查 key=A（10秒内）
→ Redis GET A = "__NULL__"
→ 直接返回空（不打 DB）✓
```

### 2.3 防击穿：互斥锁

**问题**：热点 key 过期瞬间，大量请求同时打到 DB。

**解决方案**：

```java
public PageResult<TaskListItemVO> pageTasks(...) {
    // 1. 查缓存
    PageResult<TaskListItemVO> cached = taskListCacheService.get(cacheKey);
    if (cached != null) {
        return cached;
    }
    
    // 2. 互斥锁：只让一个线程查 DB
    boolean locked = taskListCacheService.tryLock(cacheKey);
    if (!locked) {
        // 没抢到锁：短暂等待后重试
        Thread.sleep(60);
        cached = taskListCacheService.get(cacheKey);
        if (cached != null) {
            return cached;
        }
    }
    
    try {
        // 双重检查
        cached = taskListCacheService.get(cacheKey);
        if (cached != null) {
            return cached;
        }
        
        // 3. 查 DB
        PageResult<TaskListItemVO> pr = queryTasksFromDb(...);
        
        // 4. 写缓存
        taskListCacheService.set(cacheKey, pr, pr.getList() == null || pr.getList().isEmpty());
        return pr;
    } finally {
        if (locked) {
            taskListCacheService.unlock(cacheKey);
        }
    }
}
```

**互斥锁实现**：

```java
public boolean tryLock(String cacheKey) {
    // SETNX：原子设置，只有 key 不存在时才设置成功
    Boolean ok = stringRedis.opsForValue().setIfAbsent(
        LOCK_PREFIX + cacheKey, 
        "1", 
        Duration.ofSeconds(5)  // 5秒超时，防止死锁
    );
    return Boolean.TRUE.equals(ok);
}

public void unlock(String cacheKey) {
    stringRedis.delete(LOCK_PREFIX + cacheKey);
}
```

**流程**：

```
热点 key 过期瞬间：
请求A、B、C 同时到达
    ↓
A 抢到锁，查 DB，写缓存，解锁
B、C 没抢到锁，sleep 60ms，再查缓存（此时已有）→ 返回
```

### 2.4 防雪崩：随机 TTL

**问题**：大量 key 同时过期，同时打 DB。

**解决方案**：

```java
private static final int BASE_TTL = 30;
private static final int RANDOM_TTL = 30;  // 随机 0~30 秒

public void set(String cacheKey, PageResult<TaskListItemVO> data, boolean empty) {
    if (empty) {
        redis.opsForValue().set(KEY_PREFIX + cacheKey, NULL_MARK, Duration.ofSeconds(10));
    } else {
        // 随机 TTL：30~60秒，避免同时失效
        int ttl = BASE_TTL + ThreadLocalRandom.current().nextInt(RANDOM_TTL);
        redis.opsForValue().set(KEY_PREFIX + cacheKey, data, Duration.ofSeconds(ttl));
    }
}
```

**效果**：

```
原来（同时过期）：
key1: 30s → 30s后同时失效 ❌
key2: 30s → 30s后同时失效 ❌
key3: 30s → 30s后同时失效 ❌

现在（随机 TTL）：
key1: 30s → 35s后失效
key2: 45s → 47s后失效
key3: 52s → 58s后失效
（分散开，不会有雪崩）✓
```

### 2.5 三防总结

```
┌─────────────────────────────────────────────────────────┐
│                    缓存三防体系                           │
├─────────────────────────────────────────────────────────┤
│                                                          │
│  防穿透：空值缓存                                        │
│  ├─ 空结果也缓存                                        │
│  ├─ 标记 NULL_MARK                                     │
│  └─ 短 TTL（10秒）                                     │
│                                                          │
│  防击穿：互斥锁                                          │
│  ├─ tryLock 加锁                                        │
│  ├─ 只一个线程查 DB                                     │
│  └─ 其他线程短暂等待后读缓存                            │
│                                                          │
│  防雪崩：随机 TTL                                        │
│  ├─ 正常 TTL：30~60秒（随机）                          │
│  └─ 空值 TTL：10秒（固定）                              │
│                                                          │
└─────────────────────────────────────────────────────────┘
```

## 三、幂等 Token

### 3.1 什么是幂等？

**幂等**：多次执行和一次执行的结果相同。

**需要幂等的场景**：

| 场景 | 问题 | 幂等方式 |
|------|------|---------|
| 重复接单 | 一个人接多次 | 唯一索引 |
| 结算重复审核 | 审核被调多次 | 状态机 |
| 发布重复提交 | 前端重复点击 | Token |

### 3.2 发布任务的幂等场景

**问题**：用户发布任务时，可能因为网络原因点击多次，导致发布多份任务。

**解决方案**：幂等 Token

```
1. 用户进入发布页面
2. 后端生成一次性 token（UUID）
3. 前端提交时带上 token
4. 后端用 Redis DEL 原子消费 token
   - 返回 1 = 首次，合法，放行
   - 返回 0 = 已消费，重复，拒绝
```

### 3.3 代码实现

**项目代码**（[IdempotentService.java](../../server/blueblood-api/src/main/java/com/blueblood/api/common/idempotent/IdempotentService.java)）：

```java
@Service
@RequiredArgsConstructor
public class IdempotentService {
    
    private final StringRedisTemplate redis;
    private static final String PREFIX = "idem:";
    private static final Duration TTL = Duration.ofMinutes(10);
    
    // 签发 token
    public String issue(String biz) {
        String token = UUID.randomUUID().toString().replace("-", "");
        redis.opsForValue().set(PREFIX + biz + ":" + token, "1", TTL);
        return token;
    }
    
    // 消费 token（原子 DEL）
    public boolean consume(String biz, String token) {
        if (token == null || token.isEmpty()) {
            return true;  // 向后兼容：没带 token 放行
        }
        Long r = redis.execute(DEL_SCRIPT, Collections.singletonList(PREFIX + biz + ":" + token));
        return r != null && r == 1L;  // 1=首次，0=重复
    }
}
```

**Redis DEL 的原子性**：

```lua
-- DEL 返回删除的数量
-- 1 = 存在并删除（首次）
-- 0 = 不存在（已消费/过期）
return redis.call('DEL', KEYS[1])
```

### 3.4 使用方式

```java
// 1. 发布页面：获取 token
@RestController
public class TaskPublishController {
    
    @GetMapping("/task/publish/token")
    public Result<String> getPublishToken() {
        String token = idempotentService.issue("publish");
        return Result.success(token);
    }
}

// 2. 发布任务：校验 token
@PostMapping("/task/publish")
public Result<?> publishTask(
    @RequestBody PublishTaskRequest req,
    @RequestHeader(value = "X-Idempotent-Token", required = false) String token
) {
    // 幂等校验
    if (!idempotentService.consume("publish", token)) {
        throw new BusinessException(ResultCode.OPERATION_FAILED, "请勿重复提交");
    }
    
    // 发布逻辑
    taskPublishService.publish(req, token);
    return Result.success();
}
```

### 3.5 幂等方案对比

| 方案 | 适用场景 | 实现方式 |
|------|---------|---------|
| **唯一索引** | 数据库层面防重（重复接单） | 建唯一索引 |
| **状态机** | 状态流转（审核通过） | 状态判断 |
| **Token** | 无唯一约束的写操作（发布） | Redis DEL |
| **业务去重键** | 钱包流水 | (biz_type, biz_id) 查重 |

## 四、原子更新防丢失更新

### 4.1 什么是丢失更新？

**问题**：并发下"读-改-写"会丢失更新。

```
tx1: 读取 balance = 100
tx2: 读取 balance = 100
tx1: balance = 100 + 50 = 150，写回
tx2: balance = 100 + 30 = 130，写回
最终 balance = 130（tx1 的 +50 丢失了！）❌
```

### 4.2 解决方案：原子 UPDATE

**项目代码**（[WalletAccountMapper.java](../../server/blueblood-api/src/main/java/com/blueblood/api/modules/wallet/mapper/WalletAccountMapper.java)）：

```java
// 原子加余额
@Update("UPDATE wallet_account SET balance = balance + #{amount}, " +
        "total_earned = total_earned + #{amount} WHERE user_id = #{userId}")
int addBalance(@Param("userId") Long userId, @Param("amount") BigDecimal amount);

// 原子扣余额（提现冻结）
@Update("UPDATE wallet_account SET balance = balance - #{amount} " +
        "WHERE user_id = #{userId} AND balance >= #{amount}")
int deductBalance(@Param("userId") Long userId, @Param("amount") BigDecimal amount);
```

**原理**：

```
tx1: UPDATE balance = balance + 50 WHERE user_id = 1
     → balance 从 100 变成 150
     → 提交，释放锁

tx2: UPDATE balance = balance + 30 WHERE user_id = 1
     → balance 从 150 变成 180（等待 tx1 提交后再执行）
     → 提交

最终 balance = 180 ✓（两个更新都生效了）
```

### 4.3 余额不足的场景

```java

    if (affected == 0) {
        throw new BusinessException(ResultCode.OPERATION_FAILED, "余额不足");
    }

    // ...
}
```

**注意**：`WHERE balance >= #{amount}` 保证余额不会变成负数。

## 五、面试必问问题

### Q1：限流算法有哪些？你们用的哪种？

**标准回答**：
> 有固定窗口、滑动窗口、令牌桶、漏桶。我用的是固定窗口，用 Redis + Lua 实现 INCR + EXPIRE 原子计数。固定窗口最简单，但有临界突变问题；令牌桶允许突发流量。固定窗口适合验证码/登录防刷这种场景。

### Q2：缓存穿透、击穿、雪崩是什么？怎么解决？

**标准回答**：
> 穿透是查询不存在的数据，每次都打 DB，我用空值缓存解决。击穿是热点 key 过期瞬间大量请求打 DB，我用互斥锁解决。雪崩是大量 key 同时过期，我用随机 TTL 解决。

### Q3：幂等有哪些实现方式？

**标准回答**：
> 唯一索引（防重复接单）、状态机（防重复审核）、Token（防发布重复提交）、业务键去重（防结算重复入账）。我按场景选最轻量的，不滥用锁。

### Q4：为什么用原子 UPDATE 而不是先查后改？

**标准回答**：
> 因为先查后改会丢失更新。两个事务同时读到 100，各 +50 和 +30，写回时一个覆盖另一个。我用原子 UPDATE：balance = balance + ?，行锁保证更新顺序，不会丢失。
