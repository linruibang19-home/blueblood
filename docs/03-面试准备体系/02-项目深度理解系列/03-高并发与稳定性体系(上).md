## 一、为什么需要并发控制？

### 1.1 问题背景：超卖

**场景**：任务抢单（名额限量）

```
假设任务有 3 个名额，100 个用户同时抢。

❌ 朴素做法（会超卖）：
请求A：查 slots_left=3 → 判>0 → 减1 → 写回2
请求B：查 slots_left=3 → 判>0 → 减1 → 写回2
请求C：查 slots_left=3 → 判>0 → 减1 → 写回2
...（100个请求都读到3，都减1，都写回2）

结果：名额变成2，但实际接了100单 → 超卖97单！
```

### 1.2 项目中的并发场景

| 场景 | 并发问题 | 解决方案 |
|------|---------|---------|
| 任务接单 | 超卖 | Redis Lua 预扣 + DB 原子 |
| 钱包扣款 | 余额变成负数 | WHERE balance>=? |
| 浏览数++ | 丢失更新 | view_count=view_count+1 |
| 钱包入账 | 丢失更新 | balance=balance+? |
| 重复接单 | 一个人接多次 | 唯一索引 + 幂等 |

## 二、Redis Lua 秒杀预扣

### 2.1 为什么用 Redis？

| 特性 | 解释 |
|------|------|
| **单线程** | Redis 是单线程执行命令，不会被并发打断 |
| **内存操作** | QPS 高，能挡瞬时高并发 |
| **Lua 脚本** | 保证"判断+扣减"原子执行 |

### 2.2 Lua 脚本解析

**项目代码**（[TaskStockService.java](../../server/blueblood-api/src/main/java/com/blueblood/api/modules/task/service/TaskStockService.java)）：

```java
private static final DefaultRedisScript<Long> DEDUCT_SCRIPT;
static {
    DEDUCT_SCRIPT = new DefaultRedisScript<>();
    DEDUCT_SCRIPT.setScriptText(
            "local c = tonumber(redis.call('GET', KEYS[1]) or '0') " +  // 获取当前库存
            "if c > 0 then redis.call('DECR', KEYS[1]); return 1 " +    // >0 则扣减并返回1
            "else return 0 end");                                         // 否则返回0
    DEDUCT_SCRIPT.setResultType(Long.class);
}
```

**脚本逻辑**：

```
┌─────────────────────────────────────────────────────────┐
│                    Redis 单线程执行                       │
├─────────────────────────────────────────────────────────┤
│                                                         │
│  1. GET key（获取当前库存）                              │
│       ↓                                                 │
│  2. 判断 c > 0？                                        │
│       ├── Yes → DECR → return 1（成功）                  │
│       └── No  → return 0（名额已满）                    │
│                                                         │
│  ⚠️ 第1步和第2步之间不会被其他请求打断！                  │
│                                                         │
└─────────────────────────────────────────────────────────┘
```

**为什么原子？**
- Redis 单线程 → 命令顺序执行，不会被并发打断
- Lua 脚本 → 多个命令打包执行，要么全成功，要么全失败

### 2.3 懒加载机制

**问题**：Redis 重启或任务新发布时，Redis 里没有这个 key 怎么办？

**解决方案**：懒加载

```java
public boolean tryDeduct(Long taskId) {
    String k = key(taskId);
    
    // 1. 检查 Redis 有没有这个 key
    Boolean exists = redis.hasKey(k);
    if (exists == null || !exists) {
        // 2. 没有则从 DB 查 slots_left
        Task t = taskMapper.selectById(taskId);
        int slots = t == null ? 0 : (t.getSlotsLeft() == null ? 0 : t.getSlotsLeft());
        
        // 3. setIfAbsent：仅当 key 不存在时设置
        //    防止并发时多个请求同时初始化，互相覆盖
        redis.opsForValue().setIfAbsent(k, String.valueOf(slots));
    }
    
    // 4. 执行 Lua 原子预扣
    Long r = redis.execute(DEDUCT_SCRIPT, Collections.singletonList(k));
    return r != null && r == 1L;
}
```

**setIfAbsent 的作用**：
```
❌ 不用 setIfAbsent：
请求A：GET key → 不存在 → 查DB=3 → SET key=3
请求B：GET key → 不存在 → 查DB=3 → SET key=3（覆盖A的3）

✅ 用 setIfAbsent：
请求A：GET key → 不存在 → 查DB=3 → SETNX key=3 → 成功
请求B：GET key → 不存在 → SETNX key=3 → 失败（已存在，不覆盖）
```

## 三、DB 原子兜底

### 3.1 为什么需要 DB 兜底？

| 层级 | 作用 | 局限性 |
|------|------|--------|
| Redis | 挡瞬时高并发 | 内存，非持久化，可能丢失 |
| DB | 真实数据源 | QPS 低，不能单独扛高并发 |

**结论**：Redis 是缓存层，DB 才是持久化层，两者配合才能既高并发又正确。

### 3.2 原子 SQL 详解

**项目代码**（[TaskMapper.java](../../server/blueblood-api/src/main/java/com/blueblood/api/modules/task/mapper/TaskMapper.java)）：

```java
/**
 * 原子扣减剩余名额:仅当 slots_left>0 时 -1,并发安全(防超卖)。
 * 返回受影响行数:1=成功,0=名额已满。
 */
@Update("UPDATE task SET slots_left = slots_left - 1 " +
        "WHERE id = #{taskId} AND slots_left > 0 AND deleted_at IS NULL")
int decrementSlotsLeft(@Param("taskId") Long taskId);
```

**为什么能防超卖？**

```
❌ 普通做法：
UPDATE task SET slots_left = slots_left - 1 WHERE id = ?
→ 无论 slots_left 是多少都会减，可能变成负数

✅ 原子 SQL：
UPDATE task SET slots_left = slots_left - 1 
WHERE id = ? AND slots_left > 0
→ 只有 slots_left > 0 时才减，否则不执行
→ 受影响行数为 0 时说明已经没名额了
```

**行锁保证**：
```
当执行 UPDATE 时，MySQL 会对该行加排他锁（X锁）：
- 其他事务的 UPDATE 必须等待
- 只有当前事务 COMMIT 后才会释放锁

这保证了：即使两个事务同时执行，只有一个能成功减。
```

### 3.3 其他原子更新

**项目中的其他原子 SQL**：

```java
// 钱包原子加余额
@Update("UPDATE wallet_account SET balance = balance + #{amount}, " +
        "total_earned = total_earned + #{amount} WHERE user_id = #{userId}")
int addBalance(@Param("userId") Long userId, @Param("amount") BigDecimal amount);

// 钱包原子扣余额（提现冻结）
@Update("UPDATE wallet_account SET balance = balance - #{amount} " +
        "WHERE user_id = #{userId} AND balance >= #{amount}")
int deductBalance(@Param("userId") Long userId, @Param("amount") BigDecimal amount);

// 浏览数原子 +1
@Update("UPDATE task SET view_count = view_count + 1 WHERE id = #{taskId}")
int incrementViewCount(@Param("taskId") Long taskId);
```

**为什么用 `balance = balance + ?` 而不是 `balance = balance + ?`？**

这是**丢失更新**问题：

```
❌ 普通做法（会丢失更新）：
tx1: 读取 balance=100
tx2: 读取 balance=100
tx1: balance=100+50=150，写回
tx2: balance=100+30=130，写回
最终 balance=130（丢失了tx1的+50）

✅ 原子 UPDATE：
tx1: UPDATE balance=balance+50 → balance=150
tx2: UPDATE balance=balance+30 → balance=180（等待tx1完成后再执行）
最终 balance=180 ✓
```

## 四、try-catch 回补机制（重点！）

### 4.1 问题：@Transactional 为什么不回滚 Redis？

**@Transactional 的工作原理**：

```
┌─────────────────────────────────────────────────────────┐
│                  @Transactional 执行流程                │
├─────────────────────────────────────────────────────────┤
│                                                         │
│  1. 获取数据库连接                                      │
│       ↓                                                 │
│  2. 关闭自动提交（autocommit=false）                    │
│       ↓                                                 │
│  3. 执行 SQL                                           │
│       ↓                                                 │
│  4. 提交（正常）或回滚（异常）                          │
│                                                         │
│  ⚠️ 只管理数据库事务，Redis 不归它管！                   │
│                                                         │
└─────────────────────────────────────────────────────────┘
```

**问题场景**：

```java
@Transactional
public void accept(Long taskId) {
    // 1. Redis 预扣成功（名额从 3 变成 2）
    taskStockService.tryDeduct(taskId);
    
    // 2. 创建订单（成功）
    orderMapper.insert(order);
    
    // 3. DB 原子扣名额（失败！因为并发导致 slots_left 已经是 0 了）
    if (taskMapper.decrementSlotsLeft(taskId) == 0) {
        throw new BusinessException("名额已满");
    }
}
// ❌ @Transactional 检测到异常，回滚了 DB
// ❌ 但 Redis 的预扣（名额从3变2）没有被回滚！
// ❌ 结果：Redis=2，DB=3，数据不一致！
```

### 4.2 解决方案：try-catch 回补

**项目代码**（[TaskService.java:190-267](../../server/blueblood-api/src/main/java/com/blueblood/api/modules/task/service/TaskService.java)）：

```java
@Transactional
public Map<String, Long> accept(Long taskId) {
    Task t = getTask(taskId);
    
    // ... 校验逻辑 ...
    
    // 1. Redis 预扣名额
    if (!taskStockService.tryDeduct(taskId)) {
        throw new BusinessException(ResultCode.OPERATION_FAILED, "名额已满");
    }
    
    try {
        // 2. 创建订单
        TaskOrder order = new TaskOrder();
        // ...
        orderMapper.insert(order);
        
        // 3. 复制里程碑模板
        // ...
        
        // 4. DB 原子扣名额
        if (taskMapper.decrementSlotsLeft(taskId) == 0) {
            throw new BusinessException(ResultCode.OPERATION_FAILED, "名额已满");
        }
        
        return Map.of("orderId", order.getId());
        
    } catch (RuntimeException e) {
        // 5. ❗ 关键：回补 Redis！
        taskStockService.rollback(taskId);
        throw e;
    }
}
```

**rollback 方法**：

```java
/** 回补一个名额(DB 扣失败/重复接单时,把 Redis 预扣的还回去)。 */
public void rollback(Long taskId) {
    redis.opsForValue().increment(key(taskId));  // slots 从 2 变回 3
}
```

### 4.3 回补流程图

```
┌─────────────────────────────────────────────────────────────────┐
│                     try-catch 回补流程                          │
├─────────────────────────────────────────────────────────────────┤
│                                                                 │
│   用户请求                                                       │
│      │                                                          │
│      ▼                                                          │
│  Redis 预扣成功（slots: 3 → 2）                                │
│      │                                                          │
│      ▼                                                          │
│  ┌─────────────────────────────────────────┐                   │
│  │         try 块                          │                   │
│  │  ├─ 创建订单（成功）                    │                   │
│  │  ├─ 复制里程碑（成功）                  │                   │
│  │  └─ DB 原子扣名额                       │                   │
│  │       ├─ 成功 → 返回 orderId           │                   │
│  │       └─ 失败 → 抛异常                  │                   │
│  └─────────────────────────────────────────┘                   │
│      │                                                          │
│      ▼                                                          │
│  ┌─────────────────────────────────────────┐                   │
│  │         catch 块                         │                   │
│  │                                         │                   │
│  │  1. taskStockService.rollback()        │                   │
│  │     → Redis slots: 2 → 3（归还）        │                   │
│  │                                         │                   │
│  │  2. throw e（重新抛出异常）              │                   │
│  │     → @Transactional 回滚 DB             │                   │
│  └─────────────────────────────────────────┘                   │
│                                                                 │
│  最终结果：                                                      │
│  - DB 回滚（订单未创建）✓                                       │
│  - Redis 回补（名额恢复）✓                                       │
│  - 数据一致✓                                                    │
│                                                                 │
└─────────────────────────────────────────────────────────────────┘
```

### 4.4 回补的代价：最终一致性

**方案取舍**：

| 方案 | 优点 | 缺点 |
|------|------|------|
| **try-catch 回补**（项目用） | 实现简单，适合单机 Redis | 短暂不一致，可能少卖 |
| 延迟双删 | 最终一致性强 | 实现复杂，有延迟 |
| Canal 订阅 binlog | 自动同步 | 依赖外部组件 |
| 分布式事务（Seata） | 强一致 | 太重，不适合这个场景 |

**项目选择回补的原因**：
1. 单机 Redis，简单够用
2. 接受短暂不一致（少卖比超卖好）
3. DB 原子 SQL 是最后防线，不会超卖

## 六、面试必问问题

### Q1：Redis 为什么能解决超卖？

**标准回答**：
> Redis 单线程 + Lua 脚本保证"判断+扣减"原子执行，不会被并发打断。内存操作 QPS 高，能挡瞬时高并发。但 Redis 是缓存，DB 才是持久化，所以 DB 要兜底。

### Q2：为什么需要回补？

**标准回答**：
> @Transactional 只回滚 DB，回滚不了 Redis。如果预扣后 DB 失败了，Redis 预扣的名额会"漏"掉，可用名额越来越少。我用 try-catch 回补，DB 失败时手动调用 rollback 回补 Redis。

### Q3：除了回补还有什么方案？

**标准回答**：
> 延迟双删：先删缓存→更新DB→再删缓存，适合分布式。Canal 订阅 binlog：异步同步，适合多节点。分布式事务（Seata）：强一致，但太重。回补最简单，适合单机 Redis。

### Q4：DB 原子 SQL 的 WHERE slots_left>0 是必须的？

**标准回答**：
> 必须的。如果不用 WHERE 条件，UPDATE 会直接减，可能变成负数（超卖）。加了 WHERE 条件后，只有 slots_left>0 时才减，受影响行数为 0 说明已经满了。

### Q5：如果 Redis 挂了怎么办？

**标准回答**：
> Redis 挂了走 DB 兜底，只是性能下降。因为 DB 原子 SQL 是最后防线，不会超卖。只是没有了 Redis 挡流量，所有请求直接打 DB，QPS 会下降。
