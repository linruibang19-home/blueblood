## Q1：如果让你重新做，会怎么改进？

### 标准回答

> 如果让我重新做，有几个改进方向：
>
> **1. Redis 集群化**
> - 目前是单机 Redis，用 Redis Cluster 分片
> - key 的 slot = CRC16(key) % 16384
> - 扛更高并发，数据分片存储
>
> **2. 消息队列解耦**
> - 秒杀后的订单创建可以用 MQ 异步化
> - 削峰填谷，降低 DB 压力
> - 用 RabbitMQ 或 RocketMQ
>
> **3. 配置中心**
> - 目前配置写在 application.yml
> - 换成 Nacos 方便动态配置
> - 支持热更新，不用重启
>
> **4. 链路追踪**
> - 加 SkyWalking 或 Jaeger
> - 排查问题更快
> - 看到每一步的耗时
>
> **5. 灰度发布**
> - 用配置中心做流量灰度
> - 先让 10% 用户用新版本
> - 没问题再全量
>
> 这些改进需要更大规模的用户量和团队，目前单机 Redis 已经够用。

## Q2：怎么设计一个秒杀系统？

### 标准回答

> 秒杀系统设计思路：
>
> **1. 前端**
> - 按钮置灰，防重复点击
> - 前端限流（比如 3 秒只能点一次）
> - 验证码/答题分流（阿里 Paper 秒杀）
>
> **2. 接入层**
> - CDN 静态资源缓存
> - Nginx 负载均衡
> - Nginx 限流（连接数/请求数）
>
> **3. 应用层**
> - 集群部署，多台机器
> - Redis Lua 预扣库存
> - MQ 异步创建订单
> - 库存预热到 Redis
>
> **4. 数据层**
> - MySQL 原子 SQL 兜底
> - 分库分表（数据量大时）
> - 主从分离，读写分离
>
> **我的项目里**：
> - 实现了 Redis Lua 预扣 + DB 原子兜底
> - 还没用 MQ，这是可以加的方向

## Q3：如果 Redis 挂了怎么办？

### 标准回答

> 如果 Redis 挂了，分情况处理：
>
> **情况 1：Redis 挂了但服务还在**
> - 秒杀/限流：直接走 DB 兜底，只是性能下降，不会超卖
> - 缓存：直接查 DB
> - 验证码：降级成内存缓存（重启后丢失）
>
> **情况 2：Redis 挂了且服务也要重启**
> - 任务名额从 DB 懒加载，不怕数据丢失
> - 缓存数据在服务启动后重新加载
>
> **预防措施**：
> - Redis 哨兵/集群，保证高可用
> - 熔断降级，服务不可用时降级
> - 监控告警，第一时间发现

## Q4：怎么设计一个接口防刷？

### 标准回答

> 接口防刷可以从这几层入手：
>
> **1. 前端**
> - 按钮点击后置灰
> - 倒计时
>
> **2. 后端限流**
> - 固定窗口限流（我项目里用的）
> - 滑动窗口限流（更精确）
> - 令牌桶限流（允许突发）
>
> **3. 验证码**
> - 图形验证码
> - 短信/邮箱验证码
>
> **4. 行为识别**
> - 同 IP 请求频率
> - 同设备指纹
> - 异常行为检测
>
> **我项目里的实现**：
> - @RateLimit AOP 实现固定窗口限流
> - 验证码防机器注册
> - 按 IP/手机号/用户ID 多维度限流

## Q5：怎么设计一个分布式 session？

### 标准回答

> 有几种方案：
>
> **方案 1：Session 复制**
> - 多台服务器间同步 Session
> - 缺点：浪费带宽，不好扩展
>
> **方案 2：Session 集中存储**
> - Session 存 Redis
> - 所有服务器从 Redis 读写 Session
> - 缺点：依赖 Redis
>
> **方案 3：Cookie 存储**
> - Session 数据加密存在 Cookie
> - 缺点：Cookie 大小限制，安全性
>
> **方案 4：JWT（我项目里用的）**
> - Token 存在 Header，无状态
> - 服务端不存储
> - 横向扩展简单
>
> 我项目里用 JWT + Redis 存验证码，结合两者优点。

## Q6：怎么设计一个排行榜？

### 标准回答

> 排行榜设计：
>
> **方案 1：Redis ZSet**
> - `ZADD leaderboard score userId`
> - `ZREVRANGE leaderboard 0 9 WITHSCORES`
> - 自动排序，支持分页
>
> **方案 2：MySQL + 索引**
> - `SELECT * FROM user ORDER BY score DESC LIMIT 10`
> - 缺点：慢
>
> **方案 3：分桶**
> - 按分数区间分桶
> - 每个桶内用 ZSet
> - 减少数据量
>
> **实时性 vs 性能**：
> - 实时：每次更新都写 ZSet
> - 异步：批量更新，定时刷新

## Q7：怎么设计一个延时队列？

### 标准回答

> 延时队列用于订单超时关闭等场景。
>
> **方案 1：Redis ZSet + 轮询**
> - 时间戳作为 score，存入 ZSet
> - 定时任务轮询，取出到期任务执行
> - 缺点：轮询消耗资源，有延迟
>
> **方案 2：RabbitMQ 延时插件**
> - 消息设置延时时间
> - 延时到期后进入死信队列
> - 消费者处理
>
> **方案 3：RocketMQ 延时消息**
> - 支持指定延时级别
> - 延时到期后投递
>
> **方案 4：JDK DelayQueue**
> - 内存队列，需要持久化得配合其他方案
>
> 我项目里没用延时队列，订单超时关闭是后台任务定时检查。
