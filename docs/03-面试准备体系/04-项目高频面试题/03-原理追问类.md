## Q1：为什么 Redis 单线程能保证原子性？

### 标准回答

> Redis 是**单进程单线程**模型，用 epoll 做 IO 多路复用。命令在主线程顺序执行，不会被其他命令打断，所以天然原子。
>
> 但 Lua 脚本可以进一步保证多个命令的原子性——Lua 脚本里的所有命令会打包执行，要么全成功，要么全失败，不会被其他命令插入。
>
> 我项目里用 Lua 脚本实现秒杀预扣和限流，就是为了保证"判断+扣减"的原子性。

## Q2：MySQL 事务隔离级别有哪些？MySQL 默认是哪个？

### 标准回答

> 有四种：
> - **读未提交（RU）**：能看到未提交的数据，可能脏读
> - **读已提交（RC）**：只能看到已提交的数据
> - **可重复读（RR）**：MySQL 默认，事务内多次读取结果一致
> - **串行化（Serializable）**：完全串行，性能最差
>
> MySQL 用 MVCC（多版本并发控制）实现可重复读。每个事务开始时生成一个 ReadView，根据事务 ID 判断能看到哪个版本的数据。

## Q3：什么是幻读？怎么解决？

### 标准回答

> 幻读是一个事务内，两次相同查询结果不同（因为其他事务插入了新行）。
>
> 例如：
> - T1 时刻：SELECT * FROM task WHERE status='APPROVED'，结果 10 条
> - T2 时刻：另一个事务插入了 1 条 APPROVED 状态的任务
> - T3 时刻：SELECT * FROM task WHERE status='APPROVED'，结果 11 条
>
> MySQL InnoDB 在可重复读下用**间隙锁（Gap Lock）**解决幻读：锁定索引区间，不允许其他事务在这个区间插入新行。

## Q4：Redis 持久化机制是什么？

### 标准回答

> Redis 有两种持久化：
> - **RDB**：定时快照，fork 子进程生成 dump.rdb 文件。恢复快，但可能丢失最后一次快照后的数据。
> - **AOF**：记录每个写命令到 .aof 文件。数据安全，但文件大、恢复慢。
>
> Redis 4.0 支持**混合持久化**：AOF 重写时先写 RDB，再追加增量 AOF，兼顾恢复速度和安全性。
>
> 项目里没配持久化，Redis 重启数据会丢失。但任务名额从 DB 懒加载，所以不怕。

## Q5：Redis 和 MySQL 怎么保持一致？

### 标准回答

> 有几种方案：
> 1. **先删缓存再更新**：可能有短暂不一致
> 2. **先更新数据库再删缓存**（项目用）：更常用
> 3. **延迟双删**：先删→更新→再删，适合分布式
> 4. **Canal 订阅 binlog**：异步同步，适合多节点
>
> 项目里用 try-catch 回补保证一致性：Redis 预扣后如果 DB 失败，手动回补 Redis。

## Q6：什么是分布式锁？怎么实现？

### 标准回答

> 分布式锁用于多进程/多机器间的互斥访问。
>
> **Redis 实现**：
> ```lua
> -- 加锁
> SET lock_key unique_value NX PX 30000
> -- 解锁（Lua 脚本）
> if redis.call('get', KEYS[1]) == ARGV[1] then
>     return redis.call('del', KEYS[1])
> else
>     return 0
> end
> ```
>
> - NX：原子设置
> - PX：过期时间，防死锁
> - value 用唯一标识（如 UUID），解锁时校验
>
> 项目里用互斥锁防缓存击穿：`SETNX + 过期时间`

## Q7：MySQL 索引的数据结构是什么？为什么用 B+ 树？

### 标准回答

> MySQL 索引用 **B+ 树**。
>
> B+ 树是多叉平衡树，非叶子节点只存索引（主键+指针），叶子节点存完整数据（聚簇索引）或主键值（非聚簇索引），叶子节点之间用双向链表连接。
>
> 为什么用 B+ 树而不是其他结构？
> - 相比二叉树/红黑树：更矮胖，磁盘 IO 少
> - 相比 B 树：范围查询更高效（叶子节点链表）
> - 查询稳定：所有数据都在叶子节点

## Q8：什么是聚簇索引和非聚簇索引？

### 标准回答

> **聚簇索引**：叶子节点存储完整行数据。一个表只能有一个（主键索引）。
>
> **非聚簇索引**：叶子节点存储主键值。查询需要回表（根据主键再查一次）。
>
> 举例：
> - `SELECT * FROM task WHERE id = 1`（聚簇索引）→ 直接拿到数据
> - `SELECT * FROM task WHERE title = 'xxx'`（非聚簇索引）→ 先拿到 id，再查一次
>
> 覆盖索引可以避免回表：`SELECT id, title FROM task WHERE title = 'xxx'`

## Q9：Spring Bean 生命周期是什么？

### 标准回答

> Spring Bean 生命周期：
> 1. **实例化**：new 对象
> 2. **属性填充**：依赖注入（@Autowired / @RequiredArgsConstructor）
> 3. **初始化前**：`BeanPostProcessor.postProcessBeforeInitialization`
> 4. **初始化**：`@PostConstruct` / `InitializingBean.afterPropertiesSet()`
> 5. **Bean 就绪**
> 6. **销毁**：`@PreDestroy` / `DisposableBean.destroy()`
>
> 项目里用 `@PostConstruct` 做初始化，用 `@PreDestroy` 做清理。

## Q10：Spring AOP 的原理是什么？

### 标准回答

> Spring AOP 靠**代理**实现。
>
> - 有接口 → **JDK 动态代理**（接口）
> - 无接口 → **CGLIB 代理**（继承子类，重写方法）
>
> 代理对象在调用目标方法前后织入增强逻辑。
>
> 项目里的 RateLimitAspect 用 `@Around` 拦截带 `@RateLimit` 注解的方法，用 Redis + Lua 实现限流。

## Q11：什么是循环依赖？怎么解决？

### 标准回答

> 循环依赖是 A 依赖 B，B 又依赖 A。
>
> **Spring 解决循环依赖**：
> - **构造器注入**的循环依赖：无法解决，直接报错
> - **Setter 注入**的循环依赖：用**三级缓存**解决
>
> **三级缓存**：
> 1. `singletonObjects`：成品 Bean
> 2. `earlySingletonObjects`：提前暴露的 Bean（半成品）
> 3. `singletonFactories`：Bean 工厂
>
> 先创建 A，发现依赖 B，去三级缓存拿 B 的工厂，创建 B，发现依赖 A，从三级缓存拿 A 的工厂，创建 A...
>
> 项目里用 `@RequiredArgsConstructor` 构造器注入，尽量避免循环依赖。

## Q12：JVM 垃圾回收算法有哪些？

### 标准回答

> 三种基础算法：
> - **标记-清除**：标记可回收 → 清除，产生内存碎片
> - **复制**：分两半，用一半清一半，无碎片但浪费空间
> - **标记-整理**：标记 → 整理到一端 → 清除，无碎片但移动成本高
>
> JVM 用**分代收集**：
> - 年轻代：Eden + Survivor，用复制算法
> - 老年代：标记-整理算法
>
> 常见收集器：Serial、ParNew、Parallel Scavenge、Serial Old、Parallel Old、CMS、G1。
>
> 项目里用默认的 G1 收集器。
