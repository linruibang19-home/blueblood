## Q1：Redis 怎么解决超卖问题？

### 标准回答

> 任务抢单是秒杀场景，我用双层架构解决超卖：
>
> **第一层：Redis Lua 预扣**
> - Redis 单线程 + Lua 脚本，保证"判断+扣减"原子执行
> - Lua 脚本：`local c = redis.call('GET', KEYS[1]) if c > 0 then redis.call('DECR', KEYS[1]) return 1 else return 0 end`
> - 返回 1 = 预扣成功，返回 0 = 名额已满
>
> **第二层：DB 原子 SQL 兜底**
> - `UPDATE task SET slots_left = slots_left - 1 WHERE id = ? AND slots_left > 0`
> - 受影响行数为 0 说明 DB 也满了
>
> **为什么需要双层？**
> - Redis 内存操作 QPS 高，挡瞬时高并发
> - DB 是持久化层，保证数据不丢
> - 两者配合，既高并发又正确

### 追问

**Q：为什么用 Redis 做预扣而不是纯 DB？**

> 纯 DB 扛不住万级 QPS。Redis 单线程天然原子，内存操作 QPS 轻松破 10 万。但 Redis 是缓存不是持久化，挂了会丢数据，所以 DB 要兜底。

**Q：DB 原子 SQL 的 WHERE slots_left > 0 是必须的？**

> 必须的。不加 WHERE 条件，`UPDATE slots_left = slots_left - 1` 会直接减，可能变成负数。加了 WHERE 条件后，只有 slots_left > 0 才减，受影响行数为 0 说明已经满了。

## Q2：限流是怎么做的？

### 标准回答

> 我用 Redis + Lua 实现了一个 `@RateLimit` 注解 + AOP 切面。
>
> **Lua 脚本**：
> ```lua
> local c = redis.call('INCR', KEYS[1])
> if tonumber(c) == 1 then redis.call('EXPIRE', KEYS[1], ARGV[1]) end
> if tonumber(c) > tonumber(ARGV[2]) then return 0 else return 1 end
> ```
>
> - INCR 计数
> - 首次访问设过期时间
> - 超过次数返回 0（限流）
>
> **使用方式**：
> ```java
> @RateLimit(seconds = 60, count = 1, key = "#phone", message = "发送太频繁")
> public Result<?> sendCode(@RequestParam String phone) { ... }
> ```
>
> 支持 SpEL 表达式按参数限流，也支持按 IP 限流。

### 追问

**Q：限流算法有哪些？你们用的哪种？**

> 有固定窗口（我用的）、滑动窗口、令牌桶、漏桶。固定窗口最简单，有临界突变问题；令牌桶允许突发流量；滑动窗口最精确但实现复杂。

## Q3：缓存穿透、击穿、雪崩是什么？怎么解决？

### 标准回答

> **穿透**：查询不存在的数据，每次都打 DB。解决：空结果也缓存（NULL_MARK），短 TTL。
>
> **击穿**：热点 key 过期瞬间，大量请求同时打 DB。解决：互斥锁，只让一个线程查 DB。
>
> **雪崩**：大量 key 同时过期，DB 同时被打垮。解决：随机 TTL（±30秒）。

### 项目中的代码

```java
// 缓存服务
public void set(String cacheKey, PageResult data, boolean empty) {
    if (empty) {
        // 防穿透：空值短 TTL
        redis.opsForValue().set(KEY_PREFIX + cacheKey, NULL_MARK, Duration.ofSeconds(10));
    } else {
        // 防雪崩：随机 TTL
        int ttl = BASE_TTL + ThreadLocalRandom.current().nextInt(RANDOM_TTL);
        redis.opsForValue().set(KEY_PREFIX + cacheKey, data, Duration.ofSeconds(ttl));
    }
}

// 互斥锁防击穿
boolean locked = taskListCacheService.tryLock(cacheKey);
if (!locked) {
    Thread.sleep(60);  // 短暂等待
    return taskListCacheService.get(cacheKey);  // 其他线程可能已经写入
}
```

## Q4：幂等是怎么保证的？

### 标准回答

> 幂等有几种方式，我按场景选：
>
> **场景 1：重复接单**（唯一索引）
> ```sql
> UNIQUE INDEX uk_order_pair (task_id, user_id, deleted_at)
> ```
> 数据库唯一索引保证，只能有一个订单。
>
> **场景 2：发布重复提交**（Token）
> - 用户进发布页，后端生成一次性 token 存 Redis
> - 提交时带 token，后端用 Redis DEL 原子消费
> - 返回 1 = 首次，返回 0 = 重复
>
> **场景 3：钱包结算重复入账**（业务键去重）
> ```sql
> SELECT * FROM wallet_record WHERE biz_type = 'task' AND biz_id = #{milestoneId}
> ```
> 按 (biz_type, biz_id) 查重，已存在则跳过。

## Q5：钱包扣钱怎么保证不丢失更新？

### 标准回答

> 用**原子 UPDATE** 而不是"查→改→写"。
>
> **错误做法**（会丢失更新）：
> ```java
> // tx1 读到 balance=100，tx2 也读到 balance=100
> // tx1: balance=100+50=150，写回
> // tx2: balance=100+30=130，写回（覆盖了 tx1 的 50！）
> BigDecimal balance = account.getBalance();
> balance = balance.add(amount);
> account.setBalance(balance);
> mapper.updateById(account);
> ```
>
> **正确做法**（原子 UPDATE）：
> ```java
> @Update("UPDATE wallet_account SET balance = balance + #{amount} WHERE user_id = #{userId}")
> int addBalance(@Param("userId") Long userId, @Param("amount") BigDecimal amount);
> ```
>
> 行锁保证更新顺序，不会丢失。

## Q6：try-catch 回补是什么？为什么需要？

### 标准回答

> 背景：@Transactional 只回滚 DB，回滚不了 Redis。
>
> 问题：Redis 预扣后如果 DB 失败了，Redis 预扣的名额会"漏"掉，可用名额越来越少。
>
> 解决：try-catch 回补。
> ```java
> try {
>     taskStockService.tryDeduct(taskId);  // Redis 预扣
>     orderMapper.insert(order);            // DB 操作
>     taskMapper.decrementSlotsLeft(taskId);
> } catch (RuntimeException e) {
>     taskStockService.rollback(taskId);   // 回补 Redis！
>     throw e;
> }
> ```
>
> 代价是短暂不一致（少卖），但不会超卖——DB 原子 SQL 是最后防线。

### 追问

**Q：除了回补还有什么方案？**

> - 延迟双删：先删缓存→更新DB→再删缓存，适合分布式
> - Canal 订阅 binlog：异步同步，适合多节点
> - Seata 分布式事务：强一致，但太重
> - 回补最简单，适合单机 Redis

## Q7：@Transactional 为什么不回滚 Redis？

### 标准回答

> @Transactional 的原理是：
> 1. 获取数据库连接
> 2. 关闭自动提交（autocommit=false）
> 3. 执行 SQL
> 4. 提交或回滚
>
> 它只管理数据库连接，Redis 是独立的客户端，不归它管。所以 Redis 预扣后如果 DB 失败了，Redis 的状态不会被回滚。

### 追问

**Q：那 Redis 怎么保证一致性？**

> 有几种方案：回补（我用的）、延迟双删、Canal binlog、分布式事务。我选回补因为简单，适合单机 Redis。

## Q8：JWT 是怎么工作的？

### 标准回答

> 1. **登录时生成**：`JwtUtil.generate(userId, username, role)` 用 HMAC-SHA256 签名
> 2. **存储**：payload 包含 userId/username/role/过期时间
> 3. **后续请求**：带 `Authorization: Bearer xxxxx` header
> 4. **验证**：JwtAuthenticationFilter 解析 token，校验签名和过期
> 5. **注入**：存 SecurityContext，Controller 可通过 `@AuthenticationPrincipal` 获取

### 项目代码

```java
// 生成
public String generate(Long userId, String username, String role) {
    return Jwts.builder()
            .claim("userId", userId)
            .claim("username", username)
            .claim("role", role)
            .expiration(Date.from(exp))
            .signWith(key)
            .compact();
}

// 验证
public boolean isValid(String token) {
    try {
        Claims claims = parse(token);
        return claims.getExpiration().after(new Date());
    } catch (JwtException e) {
        return false;
    }
}
```
