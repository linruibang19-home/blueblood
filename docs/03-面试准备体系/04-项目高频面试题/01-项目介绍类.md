## Q2：项目最大的难点是什么？你怎么解决的？

### 标准回答

> 项目最大的难点是**高并发任务抢单**。
>
> 问题是这样的：任务抢单是秒杀场景，多个用户同时抢有限的名额。如果用朴素的"查→判→扣"，会有超卖问题——比如两个请求同时读到 slots_left=1，都判断可以扣，都减 1，结果变成 -1，这就是超卖。
>
> 我设计了一个双层架构：
> - **第一层**：Redis Lua 预扣名额。Redis 单线程 + Lua 脚本，保证"判断+扣减"原子执行，能挡瞬时高并发。
> - **第二层**：DB 原子 SQL 兜底。`UPDATE task SET slots_left = slots_left - 1 WHERE id = ? AND slots_left > 0`，受影响行数为 0 说明已经满了。
>
> 还有一个关键问题：@Transactional 只回滚 DB，回滚不了 Redis。我用 try-catch 回补：DB 失败时，手动调用 rollback 回补 Redis。虽然有短暂不一致，但不会超卖。

## Q4：项目有什么亮点？

### 标准回答（挑 2-3 个讲）

> 1. **高并发秒杀**：Redis Lua 预扣 + DB 原子兜底 + try-catch 回补
> 2. **分布式限流**：Redis + Lua 实现 @RateLimit AOP，验证码 60s/次
> 3. **缓存三防**：防穿透（空值缓存）、防击穿（互斥锁）、防雪崩（随机 TTL）
> 4. **幂等设计**：Token 防发布重复提交、钱包流水按 (biz_type, biz_id) 去重
> 5. **原子更新**：钱包余额/浏览数全部用 SQL 原子 UPDATE，防丢失更新

## Q5：项目中遇到过什么 bug？怎么解决的？

### 标准回答

> 我在项目中定位并修复了 **7+ 个真实业务 bug**，举几个典型的：
>
> 1. **提现驳回不退款的资损 bug**：原来驳回时只改了状态，没退余额。后来在驳回分支加了 `addBalance` 回补余额。
>
> 2. **退出登录无效**：`handleLogout` 方法没清 token，导致还能用旧 token 请求。后来加了清 token + store + 调后端的完整流程。
>
> 3. **并发超卖**：接单名额用"查→判→扣"，高并发下会超卖。改成 Redis Lua 预扣 + DB 原子兜底解决。
>
> 4. **进度计算卡死**：REJECTED 里程碑计入进度分母，导致进度永远到不了 100%。后来改成排除 REJECTED。
>
> 解决这些问题的方法是：加日志、分步骤打印变量、找边界条件、考虑并发场景。

## Q6：你觉得项目有什么可以改进的？

### 标准回答

> 如果让我重新做，有几个改进方向：
>
> 1. **Redis 集群化**：目前是单机 Redis，可以用 Redis Cluster 分片扛更高并发。
>
> 2. **消息队列解耦**：秒杀后的订单创建可以用 MQ 异步化，削峰填谷。
>
> 3. **配置中心**：目前配置写在 application.yml，可以换成 Nacos 方便动态配置。
>
> 4. **链路追踪**：加 SkyWalking 或 Jaeger，排查问题更方便。
>
> 5. **灰度发布**：用配置中心做流量灰度，降低发布风险。
>
> 这些改进需要更大规模的用户量和团队，目前单机 Redis 已经够用。
