## 三、Day 15-16：准备亮点讲解

### 3.1 亮点 1：Redis Lua 秒杀预扣（3 分钟）

**讲解结构**：

```markdown
## 背景

"任务抢单是一个典型的秒杀场景——多个用户同时抢有限的名额。"

"如果用朴素的'查→判→扣'，会有超卖问题。比如两个请求同时读到 slots_left=1，都判断可以扣，然后都减 1，结果 slots_left=-1，这就是超卖。"

## 方案

"我设计了一个双层架构：

第一层是 Redis Lua 预扣。Redis 单线程 + Lua 脚本，保证'判断+扣减'是原子执行的。Lua 脚本逻辑是：先 GET 看是否大于 0，如果是就 DECR 并返回成功，否则返回失败。

第二层是 DB 原子 SQL 兜底。因为 Redis 是内存缓存不是持久化，所以 DB 要作为真实数据源。DB 用的是 UPDATE task SET slots_left = slots_left - 1 WHERE id = ? AND slots_left > 0，受影响行数为 0 说明已经满了。"

## 双写一致性

"关键问题是：预扣后如果 DB 失败了怎么办？

这里有个坑：@Transactional 只回滚 DB，回滚不了 Redis。如果不处理，Redis 预扣的名额会'漏'掉，可用名额越来越少。

我的解决方案是 try-catch 回补：DB 失败时，手动调用 rollback 回补 Redis。这样虽然有短暂不一致，但不会超卖——DB 原子 SQL 是最后防线。" 
```

**必须准备的追问**：
- Q: 为什么用 Redis 做预扣而不是纯 DB？
- Q: 除了回补还有什么方案？
- Q: 固定窗口限流有什么问题？

### 3.2 亮点 2：缓存三防（2 分钟）

**讲解结构**：

```markdown
"任务大厅列表走 Redis 缓存，但缓存有三大经典问题，我都做了防护：

第一是防穿透。有些查询命中的 key 根本不存在，会反复打 DB。我的方案是：空结果也缓存，标记为 NULL_MARK，短 TTL（比如 10 秒）。

第二是防击穿。热点 key 过期瞬间，大量请求同时打到 DB。我的方案是：加互斥锁，tryLock，只让一个线程去查 DB 重建缓存，其他线程短暂等待后读缓存。

第三是防雪崩。大量 key 同时过期，缓存同时失效。我加了随机 TTL，比如正常 30~60 秒，避免同时失效。" 
```

### 3.3 亮点 3：幂等 Token（1 分钟）

**讲解结构**：

```markdown
"发布任务是防重复提交的。用户可能因为网络原因点多次，或者前端重复点击。

我的方案是幂等 Token：用户进发布页时，后端生成一个一次性 token 存在 Redis，前端提交时带这个 token。后端用 Redis DEL 原子消费——返回 1 是首次，0 是重复。

为什么不加锁？因为加锁影响性能，而且 token 去重已经够用了。" 
```
