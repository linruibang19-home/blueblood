#### 3.3.1 TaskStockService.java（Redis Lua 预扣）

**文件路径**：
```
server/blueblood-api/src/main/java/com/blueblood/api/modules/task/service/TaskStockService.java
```

**核心代码**：

```java
// Lua 脚本：GET判>0 → DECR，返回 1=成功 0=满
private static final DefaultRedisScript<Long> DEDUCT_SCRIPT;
static {
    DEDUCT_SCRIPT = new DefaultRedisScript<>();
    DEDUCT_SCRIPT.setScriptText(
            "local c = tonumber(redis.call('GET', KEYS[1]) or '0') " +
            "if c > 0 then redis.call('DECR', KEYS[1]); return 1 else return 0 end");
    DEDUCT_SCRIPT.setResultType(Long.class);
}

/**
 * 原子预扣一个名额(Lua)。
 * Redis 无 key 时(重启/未初始化)从 DB slots_left 懒加载(setIfAbsent 防并发首加载)。
 */
public boolean tryDeduct(Long taskId) {
    String k = key(taskId);
    Boolean exists = redis.hasKey(k);
    if (exists == null || !exists) {
        Task t = taskMapper.selectById(taskId);
        int slots = t == null ? 0 : (t.getSlotsLeft() == null ? 0 : t.getSlotsLeft());
        // setIfAbsent:仅当 key 不存在时设置,防并发首加载互相覆盖
        redis.opsForValue().setIfAbsent(k, String.valueOf(slots));
    }
    Long r = redis.execute(DEDUCT_SCRIPT, Collections.singletonList(k));
    return r != null && r == 1L;
}

/** 回补一个名额(DB 扣失败/重复接单时,把 Redis 预扣的还回去)。 */
public void rollback(Long taskId) {
    redis.opsForValue().increment(key(taskId));
}
```

**必须理解**：
1. **为什么用 Lua 脚本？** — Redis 单线程 + Lua 不可被打断 = 原子执行
2. **懒加载是什么？** — Redis 没有 key 时，从 DB 查 slots_left 初始化
3. **setIfAbsent 防止什么？** — 防止并发时多个请求同时初始化，互相覆盖
4. **rollback 什么时候调用？** — DB 扣失败时

// Lua 脚本：INCR + 首次 EXPIRE + 超限判定
private static final DefaultRedisScript<Long> SCRIPT;
static {
    SCRIPT = new DefaultRedisScript<>();
    SCRIPT.setScriptText(
            "local c = redis.call('INCR', KEYS[1]) " +
            "if tonumber(c) == 1 then redis.call('EXPIRE', KEYS[1], ARGV[1]) end " +
            "if tonumber(c) > tonumber(ARGV[2]) then return 0 else return 1 end");
    SCRIPT.setResultType(Long.class);
}

@Around("@annotation(rateLimit)")
public Object around(ProceedingJoinPoint pjp, RateLimit rateLimit) throws Throwable {
    // ... 解析限流 key，支持 SpEL ...
    Long allowed = redis.execute(SCRIPT, Collections.singletonList(key),
            String.valueOf(rateLimit.seconds()), String.valueOf(rateLimit.count()));
    if (allowed == null || allowed == 0L) {
        throw new BusinessException(ResultCode.OPERATION_FAILED, rateLimit.message());
    }
    return pjp.proceed();
}

1. **固定窗口限流**：INCR + EXPIRE
2. **为什么用 Lua？** — 保证 INCR 和 EXPIRE 的原子性
3. **SpEL 支持**：`#phone` / `#a0` 参数提取
4. **降级处理**：超限抛异常

/**
 * 防穿透:空结果也缓存(NULL_MARK,短 TTL),避免恶意查询反复打 DB
 * 防雪崩:正常结果随机 TTL(30~60s),避免缓存同时失效
 * 防击穿:互斥锁(tryLock),热点 key 过期时只让一个线程重建
 */

private static final String NULL_MARK = "__NULL__";
private static final int BASE_TTL = 30;
private static final int RANDOM_TTL = 30;
private static final int NULL_TTL = 10;

/** 取缓存:命中(含空值缓存)返回 PageResult,未命中返回 null。 */
public PageResult<TaskListItemVO> get(String cacheKey) {
    Object v = redis.opsForValue().get(KEY_PREFIX + cacheKey);
    if (v == null) return null;      // 未命中
    if (NULL_MARK.equals(v)) return emptyPage();  // 空值缓存，直接返回
    return (PageResult<TaskListItemVO>) v;
}

/** 写缓存:empty=true 写空值标记(短 TTL 防穿透),否则随机 TTL 防雪崩。 */
public void set(String cacheKey, PageResult<TaskListItemVO> data, boolean empty) {
    if (empty) {
        redis.opsForValue().set(KEY_PREFIX + cacheKey, NULL_MARK, Duration.ofSeconds(NULL_TTL));
    } else {
        int ttl = BASE_TTL + ThreadLocalRandom.current().nextInt(RANDOM_TTL);  // 随机 TTL
        redis.opsForValue().set(KEY_PREFIX + cacheKey, data, Duration.ofSeconds(ttl));
    }
}

/** 互斥锁防击穿 */
public boolean tryLock(String cacheKey) {
    Boolean ok = stringRedis.opsForValue().setIfAbsent(LOCK_PREFIX + cacheKey, "1", Duration.ofSeconds(5));
    return Boolean.TRUE.equals(ok);
}
```

### 3.4 必须回答的问题

#### Q1：Redis 为什么能解决超卖？

**标准回答**：
1. Redis 单线程 + Lua 脚本 → "判断+扣减"原子执行，不会被并发打断
2. 内存操作 QPS 高，能挡瞬时高并发
3. 但 Redis 是缓存，DB 才是持久化，所以 DB 要兜底

#### Q2：为什么需要回补 Redis？

**标准回答**：
1. @Transactional 只回滚 DB，回滚不了 Redis
2. 如果预扣后 DB 失败了，Redis 预扣的名额会"漏"掉
3. 可用名额越来越少，导致正常用户也接不了单

#### Q3：除了回补还有什么方案？

**标准回答**：
1. **延迟双删**：先删缓存 → 更新 DB → 再删缓存（适合分布式）
2. **Canal 订阅 binlog**：异步同步（适合多节点）
3. **分布式事务**：Seata（太重，不适合这个场景）
4. **回补最简单**，适合单机 Redis 场景

#### Q4：限流算法有哪些？你们用的哪种？

**标准回答**：
1. **固定窗口**（项目用的）：简单，有临界突变问题
2. **滑动窗口**：更精确，Redis 有序集合实现
3. **令牌桶**：允许突发流量，漏桶：平滑流出

#### Q5：缓存三防分别是什么？

**标准回答**：
1. **防穿透**：空结果也缓存（NULL_MARK，短 TTL）
2. **防击穿**：互斥锁，热点 key 过期时只让一个线程重建
3. **防雪崩**：随机 TTL，避免同时失效

### 4.3 必须回答的问题

#### Q1：JWT 怎么生成和验证？

**标准回答**：
1. **生成**：`JwtUtil.generate(userId, username, role)` → JJWT 库签名
2. **存储**：payload 包含 userId/username/role/过期时间
3. **验证**：`JwtUtil.isValid(token)` → 校验签名 + 检查过期
4. **解析**：`JwtUtil.toLoginUser(token)` → 还原 LoginUser

#### Q2：JWT 存在哪里？为什么？

**标准回答**：
1. 存在 HTTP Header：`Authorization: Bearer xxxxx`
2. 不存在 cookie（防 CSRF）
3. 前端存储在 localStorage，每次请求带 header

#### Q3：无状态 vs 有状态的区别？

**标准回答**：
- **有状态**：Session 存在服务端，分布式时需要 Session 同步
- **无状态**：Token 包含用户信息，服务端不需要存储
- **JWT 优缺点**：
  - 优点：横向扩展简单，服务端无状态
  - 缺点：Token 一旦签发无法撤销（除非加黑名单）

// 每个查询都手动加 .isNull(deletedAt)

    .isNull(Task::getDeletedAt)  // 手动过滤
    .orderByDesc(Task::getCreatedAt);
```

**好处**：
1. 可控可预期，不会被全局配置影响
2. 避免 MP 版本升级带来的兼容性问题
