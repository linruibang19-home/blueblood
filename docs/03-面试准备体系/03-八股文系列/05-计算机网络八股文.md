## 一、TCP 三次握手四次挥手

### 1.1 三次握手

```
┌─────────┐                                    ┌─────────┐
│  客户端  │                                    │  服务器  │
└────┬────┘                                    └────┬────┘
     │                                              │
     │  1. SYN=1, seq=x                          │
     │─────────────────────────────────────────────►│  Client 发送 SYN
     │                                              │
     │  2. SYN=1, ACK=1, seq=y, ack=x+1           │
     │◄─────────────────────────────────────────────│  Server 发送 SYN+ACK
     │                                              │
     │  3. SYN=0, ACK=1, seq=x+1, ack=y+1         │
     │─────────────────────────────────────────────►│  Client 发送 ACK
     │                                              │
     │              ✅ 建立连接                      │
     │                                              │
```

**为什么需要三次？**
- 第一次握手：服务器确认"我能收到消息"
- 第二次握手：客户端确认"我能发你能收" + "你能收我能发"
- 第三次握手：服务器确认"客户端能收我能发"
- **至少三次才能双方都确认对方的收发能力**

### 1.2 四次挥手

```
┌─────────┐                                    ┌─────────┐
│  客户端  │                                    │  服务器  │
└────┬────┘                                    └────┬────┘
     │                                              │
     │  1. FIN=1, seq=u                           │
     │─────────────────────────────────────────────►│  Client 发 FIN
     │                                              │
     │  2. ACK=1, ack=u+1                          │
     │◄─────────────────────────────────────────────│  Server 回 ACK
     │     ← 此时：Server 还能发，Client 不能再发 →  │
     │                                              │
     │  ... (Server 可能还有数据要发) ...           │
     │                                              │
     │  3. FIN=1, seq=w                            │
     │◄─────────────────────────────────────────────│  Server 发 FIN
     │                                              │
     │  4. ACK=1, ack=w+1                         │
     │─────────────────────────────────────────────►│  Client 回 ACK
     │                                              │
     │  Client 等待 2MSL                            │
     │                                              │
     │              ✅ 关闭连接                      │
     │                                              │
```

**为什么需要四次？**
- Client 发 FIN：Client 不再发送，但 Server 可能还有数据要发
- Server 发 ACK：确认收到，但连接还没关闭
- Server 发 FIN：Server 也没数据了
- Client 发 ACK：确认收到

**为什么要等 2MSL？**
- 确保 Server 能收到最后的 ACK
- 让本连接的报文在网络中消失，不影响后续连接

## 二、TCP 可靠传输

### 2.1 可靠传输机制

| 机制 | 说明 | 项目中的体现 |
|------|------|------------|
| **序号/确认** | 每个字节都有序号，收到后要确认 | HTTP 请求/响应 |
| **超时重传** | 超时没收到确认就重传 | TCP 协议栈实现 |
| **流量控制** | 滑动窗口，不让发送方发太快 | 接收方 buffer |
| **拥塞控制** | 慢启动、拥塞避免、快速恢复 | 网络拥塞时 |

### 2.2 TCP vs UDP

| 维度 | TCP | UDP |
|------|-----|-----|
| 连接 | 面向连接 | 无连接 |
| 可靠性 | 可靠 | 不可靠 |
| 传输速度 | 慢 | 快 |
| 报文 | 字节流 | 数据报 |
| 头部 | 20-60 字节 | 8 字节 |
| 场景 | HTTP、文件传输、邮件 | 视频、语音、DNS |

## 三、HTTP 协议

### 3.1 HTTP 1.0 vs 1.1 vs 2.0 vs 3.0

| 版本 | 关键特性 | 缺点 |
|------|---------|------|
| **HTTP 1.0** | 短连接，每请求重建连接 | 连接开销大 |
| **HTTP 1.1** | 长连接（keep-alive）、管道化 | 队头阻塞 |
| **HTTP 2.0** | 多路复用、头部压缩、Server Push | TCP 队头阻塞 |
| **HTTP 3.0** | QUIC（UDP）、无队头阻塞 | 依赖 UDP |

### 3.2 HTTP 常用状态码

| 状态码 | 含义 | 例子 |
|--------|------|------|
| **200** | 成功 | `{"code":0,"data":{}}` |
| **400** | 请求参数错误 | 参数校验失败 |
| **401** | 未认证 | 没带 token / token 过期 |
| **403** | 无权限 | 不是 ADMIN 访问后台 |
| **404** | 资源不存在 | 数据被删了 |
| **500** | 服务器内部错误 | NPE / 空指针 |
| **502** | 网关错误 | 后端服务挂了 |
| **503** | 服务不可用 | 限流/熔断 |

### 3.3 HTTP 请求方法

| 方法 | 说明 | 项目中的例子 |
|------|------|------------|
| **GET** | 查询 | `GET /task/1` |
| **POST** | 创建 | `POST /task/accept` |
| **PUT** | 更新 | `PUT /user/profile` |
| **DELETE** | 删除 | `DELETE /post/1` |

## 四、HTTPS

### 4.1 HTTPS 工作原理

```
┌─────────┐                    ┌─────────┐                    ┌─────────┐
│  客户端  │                    │  服务器  │                    │   CA    │
└────┬────┘                    └────┬────┘                    └────┬────┘
     │                              │                              │
     │  1. Client Hello（支持的加密套件）                         │
     │──────────────────────────────►│                              │
     │                              │                              │
     │  2. Server Hello + 证书 + 公钥                            │
     │◄─────────────────────────────│                              │
     │                              │                              │
     │  3. 验证证书（查 CA 公钥）                                 │
     │─────────────────────────────────────────────────────────────►│
     │                              │                              │
     │  4. 发送 Pre-Master Secret（用公钥加密）                  │
     │──────────────────────────────►│                              │
     │                              │                              │
     │  双方用 Pre-Master 生成对称密钥     │                      │
     │                              │                              │
     │  5. 之后用对称密钥加密通信 ✅    │                          │
     │◄─────────────────────────────►│                          │
     │                              │                              │
```

### 4.2 对称加密 vs 非对称加密

| 类型 | 原理 | 优点 | 缺点 |
|------|------|------|------|
| **对称加密** | 同一把密钥加解密 | 快 | 密钥传输不安全 |
| **非对称加密** | 公钥加密私钥解密 | 安全 | 慢 |

**HTTPS 做法**：
1. 用非对称加密传对称密钥（解决密钥传输问题）
2. 之后用对称密钥通信（快）

## 五、Cookie vs Session

| 维度 | Cookie | Session |
|------|--------|---------|
| 存储位置 | 浏览器 | 服务器 |
| 安全性 | 较低（可被伪造） | 较高 |
| 大小限制 | 4KB | 无限制 |
| 生命周期 | 可设置过期时间 | 跟随会话 |
| 占用服务器资源 | 不占 | 占 |

**项目中用 JWT 而不是 Session**：
- JWT 存在 Header，不占 Cookie 空间
- JWT 无状态，服务器不存储
- 适合多端（前后端分离 + 小程序）

## 六、面试必问问题

### Q1：TCP 三次握手四次挥手是什么？为什么？

**标准回答**：
> 三次握手是建立连接的过程：客户端发 SYN，服务器回 SYN+ACK，客户端再回 ACK。三次才能让双方都确认收发能力正常。四次挥手是关闭连接：客户端发 FIN，服务器回 ACK，服务器发 FIN，客户端回 ACK。因为 TCP 是全双工，关闭时需要双向确认，所以至少四次。

### Q2：TCP 和 UDP 的区别？

**标准回答**：
> TCP 面向连接、可靠传输、速度慢，适合 HTTP、文件传输。UDP 无连接、不可靠传输、速度快，适合视频、语音、DNS。项目里 HTTP 请求用的是 TCP，实时性要求高的场景可能用 UDP。

### Q3：HTTP 和 HTTPS 的区别？

**标准回答**：
> HTTP 是明文传输，不安全。HTTPS 用 TLS/SSL 加密传输，更安全。HTTPS 用了非对称加密传对称密钥，之后用对称密钥加密通信。项目中用 HTTPS 需要配证书。

### Q4：Cookie 和 Session 的区别？

**标准回答**：
> Cookie 存在浏览器，大小限制 4KB，安全性较低。Session 存在服务器，安全性高，但占服务器资源。项目中用 JWT 而不是 Session，因为 JWT 无状态适合前后端分离多端场景。
