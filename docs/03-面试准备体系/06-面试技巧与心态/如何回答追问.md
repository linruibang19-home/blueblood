# 如何回答追问

> 应对面试官追问技巧

---

## 一、追问类型分类

面试官的追问通常分这几类：

| 类型 | 特点 | 应对策略 |
|------|------|---------|
| **技术深挖** | 问原理、为什么 | 讲清原理，准备追问预案 |
| **方案对比** | 问其他方案、优缺点 | 提前准备方案对比 |
| **场景应用** | 问如果XXX怎么办 | 展示思考过程 |
| **细节确认** | 问代码、参数、配置 | 回答准确，不要瞎编 |

---

## 二、应对技巧

### 1. 技术深挖类

**原则**：讲清原理，不要只说结论

❌ 不好：
```
面试官：Redis 怎么保证原子性？
回答：Redis 单线程，所以能保证原子性。
```

✅ 好：
```
面试官：Redis 怎么保证原子性？
回答：Redis 是单进程单线程模型，用 epoll 做 IO 多路复用。
命令在主线程顺序执行，不会被其他命令打断，所以天然原子。
但如果是多个命令需要原子，可以用 Lua 脚本打包执行，
Lua 脚本里的所有命令会作为一个整体执行，要么全成功要么全失败。
我项目里用 Lua 脚本实现秒杀预扣，就是为了保证"判断+扣减"的原子性。
```

### 2. 方案对比类

**原则**：提前准备对比，说清取舍

❌ 不好：
```
面试官：除了回补还有什么方案？
回答：还有延迟双删、Canal、分布式事务。
```

✅ 好：
```
面试官：除了回补还有什么方案？
回答：有几种方案，各有取舍：
1. 延迟双删：先删缓存→更新DB→再删缓存，适合分布式场景。
   缺点是有延迟，实现稍复杂。
2. Canal 订阅 binlog：异步同步数据，适合多节点。
   缺点是依赖外部组件。
3. 分布式事务（Seata）：强一致，但太重了。
   缺点是性能开销大，不适合高并发场景。

我选回补是因为简单，适合单机 Redis 的场景。单机下回补已经够用，
而且 DB 原子 SQL 是最后防线，不会超卖。
```

### 3. 场景应用类

**原则**：展示思考过程，不要只给答案

❌ 不好：
```
面试官：如果 Redis 挂了怎么办？
回答：走 DB 兜底。
```

✅ 好：
```
面试官：如果 Redis 挂了怎么办？
回答：分情况处理：

1. Redis 挂了但服务还在：
   - 秒杀/限流：直接走 DB 兜底，只是性能下降，不会超卖
   - 缓存：直接查 DB
   - 验证码：降级成内存缓存（重启后丢失，但验证码本来就是临时的）

2. 如果担心 Redis 挂了：
   - 可以加 Redis 哨兵/集群，保证高可用
   - 可以加熔断降级，服务不可用时降级
   - 可以加监控告警，第一时间发现

我项目里因为任务名额从 DB 懒加载，所以 Redis 重启数据丢失也不怕。
```

### 4. 细节确认类

**原则**：诚实回答，不知道就说不知道

❌ 不好：
```
面试官：这个配置项的值是多少？
回答：应该是...（不确定）
```

✅ 好：
```
面试官：这个配置项的值是多少？
回答：具体的值我记不太清了，但我记得大概范围是 XXX。
这个配置是在 application.yml 里配的，当时调优的时候根据压测结果
调整的。如果需要我可以回去查一下。
```

## 三、常见追问及标准回答

### Q1：为什么用 Redis 而不是纯 DB？

**标准回答**：
> Redis 单线程 + Lua 脚本保证原子，内存操作 QPS 高，能挡瞬时高并发。
> 但 Redis 是缓存不是持久化，挂了可能丢数据，所以 DB 要兜底。
> 两者配合，既高并发又正确。

### Q2：@Transactional 为什么不回滚 Redis？

**标准回答**：
> @Transactional 只管理数据库连接，Redis 是独立的客户端不归它管。
> Redis 预扣后如果 DB 失败了，Redis 的状态不会被回滚。
> 我用 try-catch 回补解决。

### Q3：限流算法有哪些？你们用的哪种？

**标准回答**：
> 有固定窗口（我用的）、滑动窗口、令牌桶、漏桶。
> 固定窗口最简单，有临界突变问题；令牌桶允许突发流量。
> 我用固定窗口是因为验证码/登录防刷场景，固定窗口够用且简单。

### Q4：除了这些还有其他方案吗？

**标准回答**：
> 方案有很多，我按场景选的：
> - Redis 做缓存：单机够用，大规模用 Redis Cluster
> - 分布式锁：Redis SETNX + 唯一值 + 过期时间
> - 延迟队列：RabbitMQ 延时插件 / RocketMQ 延时消息

### Q5：如果让你重新做，会怎么改进？

**标准回答**：
> 如果重新做，有几个改进方向：
> 1. Redis 集群化（用 Cluster 分片）
> 2. MQ 异步化（削峰填谷）
> 3. 配置中心（Nacos）
> 4. 链路追踪（SkyWalking）
> 这些需要更大规模的用户量和团队，目前单机已经够用。

## 四、不会回答怎么办？

### 1. 诚实承认

```
"这个我没深入了解过，不过根据我的理解..."
```

### 2. 展示思考

```
"这个问题我没遇到过，但如果是我的话，我会先排查...
然后根据情况..."
```

### 3. 引导到熟悉的领域

```
"这个问题跟 XX 类似，区别是...我在项目里是这样处理的..."
```

### 4. 追问确认

```
"您是指 XX 场景吗？我理解的..."
```

---

## 五、追问后怎么收尾？

回答完追问后，可以加一句：

- "不知道我讲清楚了吗？"
- "这个点您想深入了解吗？"
- "我可能讲得不够清楚，您想从哪个角度再问一下？"

这样既展示了自信，又给了面试官继续追问的机会。
