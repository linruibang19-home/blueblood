# 05 · 业务全流程从用户视角走一遍

> 目标：先像真实用户一样用项目，而不是一上来读代码。

---

## 为什么要从用户视角开始？

因为代码只是业务的实现。如果你不知道用户在页面上做了什么，就很难理解后端为什么要这么设计。

正确顺序：

```text
先走页面 → 再理解业务对象 → 再看接口 → 再读代码 → 最后看表
```

---

## 路线 A：普通用户完整路径

适合账号：

```text
lin / 123456
```

### 1. 登录

用户操作：

```text
打开前台 H5 → 输入账号密码 → 登录成功 → 进入首页/我的页面
```

背后发生：

```text
POST /auth/login
→ 后端校验用户名密码
→ 签发 JWT token
→ 前端保存 token
→ 后续请求带 Authorization: Bearer xxx
```

### 2. 浏览任务大厅

用户操作：

```text
点击“任务”Tab → 查看任务列表 → 按分类/关键词筛选
```

背后发生：

```text
GET /task/page
→ 后端先查 Redis 缓存
→ 缓存没有再查 MySQL
→ 写回 Redis 缓存
```

### 3. 查看任务详情

用户操作：

```text
点击一个任务 → 查看任务标题、描述、金额、技能、名额、里程碑
```

背后发生：

```text
GET /task/{id}
→ 查询任务信息
→ 查询任务技能
→ 查询里程碑模板
→ 浏览数 view_count 原子 +1
```

### 4. 接单

用户操作：

```text
点击“接单”按钮 → 接单成功 → 跳转到任务执行页
```

背后发生：

```text
POST /task/{id}/accept
→ 校验任务状态
→ 校验是否重复接单
→ 校验等级
→ Redis Lua 原子预扣名额
→ 创建 TaskOrder
→ 复制里程碑模板为订单实例
→ DB 原子扣减 slots_left
→ 返回 orderId
```

这是项目最核心的链路之一。

### 5. 查看我的任务

用户操作：

```text
进入“我的任务” → 查看已接订单 → 点击任务执行
```

背后发生：

```text
GET /task/my-orders
→ 根据当前登录用户 userId 查询 task_order
→ 拼装任务信息和里程碑信息
```

### 6. 提交里程碑

用户操作：

```text
选择一个里程碑 → 填写成果说明/GitHub 地址/附件 → 提交
```

背后发生：

```text
POST /task/milestones/{id}/submit
→ 校验里程碑归属
→ 保存 MilestoneSubmission
→ 里程碑状态改为 SUBMITTED
```

### 7. 查看钱包

用户操作：

```text
进入“我的” → “钱包” → 查看余额、收益流水、提现入口
```

背后发生：

```text
GET /wallet/account
GET /wallet/records
```

如果里程碑审核通过，收益会出现在钱包流水中。

## 路线 B：企业用户完整路径

企业用户和普通用户最大的区别：

```text
企业用户可以发布任务，并作为雇主审核里程碑。
```

### 1. 企业认证

用户操作：

```text
注册企业账号 → 填写企业资料 → 提交认证申请
```

背后发生：

```text
POST /enterprise/application
→ 创建 enterprise_application
→ 状态 PENDING
```

### 2. 管理员审核企业

管理员操作：

```text
后台 → 企业认证审核 → APPROVED
```

背后发生：

```text
admin 审核通过
→ user.user_type 从 personal 改为 enterprise
```

### 3. 发布任务

企业用户操作：

```text
进入任务发布页 → 填写任务信息 → 配置多个里程碑 → 提交
```

背后发生：

```text
获取幂等 token
→ 校验 user_type=enterprise
→ 校验 reward = 里程碑金额之和
→ 创建 task
→ 创建 task_skill
→ 创建 task_milestone 模板
→ 初始化 Redis 库存 task:slots:{taskId}
→ 清任务列表缓存
```

### 4. 审核里程碑

企业用户操作：

```text
进入我发布的任务 → 查看接单订单 → 查看里程碑提交 → APPROVED/REJECTED
```

背后发生：

```text
POST /task/milestones/{id}/review
→ 校验当前用户是否是任务 employer
→ 写 milestone_review
→ 改里程碑状态
→ APPROVED 时触发钱包入账
```

## 路线 D：学习成长路径

普通用户操作：

```text
进入“成长”Tab
→ 浏览课程列表
→ 进入课程详情
→ 学习章节
→ 更新学习进度
→ 提交作业
→ 查看作业结果
```

背后发生：

```text
GET /course/page
GET /course/{id}
POST /course/{id}/progress
POST /assignment/submit
```

学习进度模型：

```text
completed_chapters = max(原值, 当前完成章节序号)
progress = completed_chapters / total_chapters
```

特点：只前进，不回退。

---

## 四条路线的关系

```text
普通用户路线：接单赚钱
企业用户路线：发布任务、审核交付
管理员路线：平台运营和兜底审核
学习路线：提升技能，为接任务服务
```

它们最终都服务于主闭环：

```text
学习成长 → 实战接单 → 交付验收 → 钱包收益
```
