# 12 · 核心链路二：任务发布与接单

> 目标：理解项目最核心的业务链路：企业发布任务，用户接单，系统生成订单和里程碑实例。

---

## 一、为什么这条链路最重要？

蓝血菁英平台的核心价值是：

```text
让企业/雇主发布任务，让 AI 技术人才接单交付并获得收益。
```

所以最重要的链路就是：

```text
发布任务 → 初始化库存 → 用户接单 → 生成订单 → 复制里程碑 → 扣减名额
```

这条链路同时包含：

```text
业务建模 + 并发控制 + Redis + MySQL + 幂等 + 事务
```

---

## 二、任务发布链路

### 1. 用户操作

企业用户在前台填写：

```text
任务标题
任务描述
总金额 reward
任务名额 total_slots
技能要求 skills
里程碑列表 milestones
```

每个里程碑包含：

```text
标题
描述
截止时间
金额 reward
```

### 2. 后端链路

```text
TaskPublishController
→ TaskPublishService.publish(req, token)
→ 校验幂等 token
→ 校验当前用户是否 enterprise
→ 校验任务总金额 = 里程碑金额之和
→ 创建 task
→ 创建 task_skill
→ 创建 task_milestone 模板
→ TaskStockService.initStock(taskId, slots)
→ TaskListCacheService.evictAll()
```

### 3. 为什么发布要幂等？

用户可能：

```text
网络卡顿重复点击
浏览器重试
前端重复提交
```

如果不防重，可能发布出两条一模一样的任务。

项目用：

```text
Redis 一次性幂等 token
```

流程：

```text
进入发布页 → 后端签发 token → 提交时带 X-Idempotent-Token → Redis DEL 原子消费
```

## 三、任务接单链路总览

用户点击接单后：

```text
POST /task/{id}/accept
→ TaskController.accept
→ TaskService.accept
```

核心步骤：

```text
1. 查询任务
2. 校验任务状态
3. 校验是否重复接单
4. 校验用户等级
5. Redis Lua 预扣名额
6. 创建 task_order
7. 复制 task_milestone 模板为订单实例
8. DB 原子扣减 slots_left
9. 如果 DB 失败，回补 Redis
10. 返回 orderId
```

---

## 四、为什么接单像秒杀？

因为任务名额有限。

例如：

```text
一个任务只有 3 个名额，100 个用户同时点击接单。
```

如果简单写：

```text
先查 slots_left
如果 > 0
再 slots_left - 1
```

就会超卖。

所以项目用了两层防线：

```text
第一层：Redis Lua 原子预扣
第二层：MySQL 原子 UPDATE 兜底
```

---

## 五、Redis Lua 预扣

代码位置：

```text
TaskStockService.tryDeduct(taskId)
```

逻辑：

```text
1. Redis key = task:slots:{taskId}
2. 如果 key 不存在，从 DB slots_left 懒加载
3. 执行 Lua 脚本
4. 如果库存 > 0，DECR 并返回 1
5. 如果库存 <= 0，返回 0
```

Lua 逻辑：

```lua
local c = tonumber(redis.call('GET', KEYS[1]) or '0')
if c > 0 then
  redis.call('DECR', KEYS[1])
  return 1
else
  return 0
end
```

为什么原子？

```text
Redis 单线程 + Lua 脚本不可被其他命令打断。
```

## 六、DB 原子扣减

代码位置：

```text
TaskMapper.decrementSlotsLeft(taskId)
```

SQL：

```sql
UPDATE task
SET slots_left = slots_left - 1
WHERE id = ?
  AND slots_left > 0
  AND deleted_at IS NULL
```

为什么还要 DB 扣？

```text
Redis 是缓存层，MySQL 才是真实数据源。
Redis 负责挡高并发，DB 负责保证最终正确。
```

如果返回影响行数 0：

```text
说明 DB 里名额也没了，必须失败。
```

---

## 七、为什么要回补 Redis？

接单流程里 Redis 先扣，DB 后扣。

如果 Redis 扣成功，但后面失败了，例如：

```text
订单唯一索引冲突
DB slots_left 已经是 0
插入里程碑失败
其他运行时异常
```

那么 DB 事务会回滚，但是 Redis 不会自动回滚。

原因：

```text
@Transactional 只管理数据库连接，不管理 Redis。
```

所以代码里需要：

```java
try {
    // 创建订单、复制里程碑、DB扣名额
} catch (RuntimeException e) {
    taskStockService.rollback(taskId);
    throw e;
}
```

回补逻辑：

```text
INCR task:slots:{taskId}
```

## 八、任务和订单的关系

发布任务时：

```text
task 创建一条任务
task_milestone 创建里程碑模板，order_id = NULL
```

接单时：

```text
task_order 创建一条订单
task_milestone 复制模板，order_id = 当前订单 ID
```

所以：

```text
模板属于 task；
实例属于 task_order。
```

---

## 九、接单成功后的结果

用户接单成功后，系统里会发生：

```text
1. task_order 新增一条记录
2. task_milestone 新增若干订单实例
3. task.slots_left 减 1
4. Redis task:slots:{taskId} 减 1
5. 如果 slots_left 归零，任务可能进入 IN_PROGRESS
6. 前端拿到 orderId 跳转任务执行页
```

## 十、这条链路怎么排错？

### 接单提示“名额已满”

可能原因：

```text
Redis 库存为 0
DB slots_left 为 0
Redis 和 DB 不一致但 DB 兜底失败
```

### 接单提示“你已接取该任务”

可能原因：

```text
hasOrder 查询发现已有订单
唯一索引 uk_order_pair 冲突
```

### 接单后没有 orderId

可能原因：

```text
后端返回结构变化
前端没有正确解包 Result<T>
接口异常但前端没提示
```
