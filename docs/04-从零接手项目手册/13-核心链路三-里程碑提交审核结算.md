# 13 · 核心链路三：里程碑提交、审核、结算

> 目标：理解接单后怎么交付、雇主怎么审核、钱包什么时候入账。

---

## 一、这条链路解决什么问题？

用户接单之后，不能直接拿钱。必须：

```text
按里程碑提交成果 → 雇主审核通过 → 系统按里程碑金额入账钱包
```

这样能解决：

```text
1. 大任务可以分阶段交付
2. 雇主可以逐阶段验收
3. 接单者可以逐阶段拿钱
4. 平台可以避免重复结算
```

---

## 二、链路总览

```text
接单者提交里程碑
→ 保存 milestone_submission
→ 里程碑状态变为 SUBMITTED
→ 雇主审核 APPROVED/REJECTED
→ APPROVED 触发 WalletService.credit
→ 钱包流水幂等查重
→ 原子加余额
→ 写 wallet_record
→ 重新计算订单进度
→ 全部通过则订单 settled
```

---

## 三、提交里程碑

### 用户操作

```text
进入任务执行页
→ 选择某个里程碑
→ 填写成果说明
→ 填 GitHub 地址/附件
→ 点击提交
```

### 后端链路

```text
POST /task/milestones/{id}/submit
→ MilestoneService.submit(milestoneId, req)
→ 校验当前用户是否订单接单者
→ 校验状态是否允许提交
→ 保存或更新 milestone_submission
→ task_milestone.status = SUBMITTED
→ review_status = PENDING
```

### 数据变化

```text
milestone_submission 新增/更新
任务里程碑状态改为 SUBMITTED
```

## 四、审核里程碑

### 谁能审核？

```text
1. 该任务的雇主 employer
2. 管理员 ADMIN（后台兜底）
```

普通接单者不能审核自己的里程碑。

### 审核结果

| 状态 | 含义 | 后续动作 |
|---|---|---|
| APPROVED | 审核通过 | 触发钱包入账 |
| REJECTED | 审核驳回 | 不入账，接单者可重新提交 |

### 后端链路

```text
POST /task/milestones/{id}/review
→ MilestoneService.review(milestoneId, req)
→ 校验审核人是否 employer 或 ADMIN
→ 写 milestone_review
→ 修改 task_milestone.status
→ 如果 APPROVED，调用 WalletService.credit
→ recomputeOrderProgress(orderId)
```

---

## 五、为什么是分阶段结算？

如果一个任务总金额 3000 元，分 3 个里程碑：

```text
需求分析：500
核心开发：1500
测试交付：1000
```

那么不是等全部完成后一次性入账 3000，而是：

```text
需求分析 APPROVED → 入账 500
核心开发 APPROVED → 入账 1500
测试交付 APPROVED → 入账 1000
```

优点：

```text
1. 接单者更早获得阶段收益
2. 雇主风险更低
3. 平台业务流程更真实
```

## 六、钱包入账怎么保证不重复？

审核接口可能被重复调用，例如：

```text
前端重复点击
网络重试
管理员重复操作
```

如果不做幂等，同一个里程碑会重复入账。

项目用业务键去重：

```text
biz_type = task
biz_id = 里程碑ID
```

逻辑：

```text
WalletService.credit(userId, amount, "task", milestoneId, remark)
→ 查询 wallet_record 是否已有 biz_type=task 且 biz_id=milestoneId 的 income 记录
→ 如果已有，直接跳过
→ 如果没有，原子加余额并写流水
```

---

## 七、钱包入账为什么要原子 UPDATE？

错误方式：

```text
先查余额 balance=100
Java 里 balance + 500
再 update balance=600
```

高并发下会丢失更新。

正确方式：

```sql
UPDATE wallet_account
SET balance = balance + ?, total_earned = total_earned + ?
WHERE user_id = ?
```

这样 MySQL 行锁会保证多个入账顺序执行。

## 八、订单进度怎么计算？

每次审核后重新计算：

```text
有效里程碑 = 排除 REJECTED 的里程碑
已通过里程碑 = APPROVED 的里程碑
progress = 已通过 / 有效里程碑 * 100
```

为什么排除 REJECTED？

```text
如果 REJECTED 也算分母，订单可能卡死，永远到不了 100%。
```

当全部有效里程碑都 APPROVED：

```text
task_order.status = settled
```

## 九、数据流总结

```text
task_order
  │
  └── task_milestone（实例）
          │
          ├── milestone_submission（接单者提交成果）
          └── milestone_review（雇主/管理员审核）
                  │
                  └── APPROVED
                          │
                          ▼
                    wallet_account 原子加余额
                          │
                          ▼
                    wallet_record 写入流水
```

## 十、这条链路怎么排错？

### 提交失败

可能原因：

```text
当前用户不是订单接单者
里程碑状态不允许提交
附件格式或字段错误
```

### 审核失败

可能原因：

```text
当前用户不是雇主
当前用户不是 ADMIN
里程碑不存在
状态不允许审核
```

### 审核通过但钱包没到账

可能原因：

```text
WalletService.credit 幂等判断发现已有流水
wallet_account 不存在
DB 原子加余额失败
事务回滚
```

### 订单进度不对

可能原因：

```text
REJECTED 是否被错误计入分母
里程碑状态大小写是否一致
订单下是否有里程碑实例
```
