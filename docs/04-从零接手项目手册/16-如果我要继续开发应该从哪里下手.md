# 16 · 如果我要继续开发应该从哪里下手

> 目标：以后要加新功能或改 bug 时，知道应该改哪些层、看哪些文件。

---

## 一、先判断你要改哪一类功能

| 需求类型 | 主要改动位置 |
|---|---|
| 后端新增接口 | `server/blueblood-api` |
| 前台用户页面 | `apps/user-web` |
| 后台管理页面 | `apps/admin-web` |
| 小程序页面 | `apps/miniapp` |
| 数据库字段/表 | `sql/` + 后端 Entity/Mapper |
| 部署配置 | `deploy/` |
| 文档 | `docs/` |

---

## 二、新增一个后端功能的标准流程

假设要新增“任务评价”功能。

推荐顺序：

```text
1. 设计数据表
2. 写 SQL 迁移
3. 创建 Entity
4. 创建 Mapper
5. 创建 Request/VO DTO
6. 写 Service 业务逻辑
7. 写 Controller 接口
8. 前端 api 封装
9. 前端页面接入
10. 后台管理兜底
11. 更新文档
```

---

## 三、后端新增模块模板

目录结构建议：

```text
server/blueblood-api/src/main/java/com/blueblood/api/modules/review/
├── controller/
│   └── TaskReviewController.java
├── service/
│   └── TaskReviewService.java
├── mapper/
│   └── TaskReviewMapper.java
├── entity/
│   └── TaskReview.java
└── dto/
    ├── CreateReviewRequest.java
    └── TaskReviewVO.java
```

命名规则：

```text
Controller：接口入口
Service：业务逻辑
Mapper：数据库访问
Entity：表映射
DTO/VO：请求和响应对象
```

## 四、如果要改任务模块

优先看：

```text
TaskController.java
TaskService.java
TaskPublishService.java
MilestoneService.java
TaskStockService.java
TaskListCacheService.java
TaskMapper.java
```

常见改动：

| 改动 | 看哪里 |
|---|---|
| 任务列表筛选 | `TaskService.pageTasks` |

| 接单逻辑 | `TaskService.accept` |

| 里程碑提交 | `MilestoneService.submit` |
| 里程碑审核 | `MilestoneService.review` |
| 秒杀库存 | `TaskStockService` |
| 任务列表缓存 | `TaskListCacheService` |

## 六、如果要改认证权限

优先看：

```text
SecurityConfig.java
JwtAuthenticationFilter.java
JwtUtil.java
SecurityUtils.java
AuthService.java
```

常见改动：

| 改动 | 看哪里 |
|---|---|
| 放行新接口 | `SecurityConfig` |
| token 字段 | `JwtUtil.generate` / `toLoginUser` |
| 当前用户获取 | `SecurityUtils` |

| 注册方式 | `AuthService.register` |

## 九、改动前的检查清单

每次动手前先问自己：

```text
[ ] 这个需求属于哪个业务域？
[ ] 需要新增表吗？
[ ] 需要新增接口吗？
[ ] 需要前台页面吗？
[ ] 需要后台管理吗？
[ ] 是否涉及权限？
[ ] 是否涉及钱包金额？
[ ] 是否涉及并发？
[ ] 是否需要 Redis？
[ ] 是否需要更新文档？
```

---

## 十、新功能开发推荐顺序

标准顺序：

```text
SQL
→ Entity
→ Mapper
→ DTO
→ Service
→ Controller
→ 前端 API
→ 页面
→ 测试
→ 文档
```

如果涉及钱包/库存/审核，额外注意：

```text
幂等
事务
权限
状态机
并发安全
```

---

## 十一、继续开发时最容易出错的点

```text
1. 忘记 deleted_at IS NULL 过滤软删数据
2. 忘记 @PreAuthorize 限制后台接口
3. 钱包金额用普通读改写，导致并发丢失更新
4. Redis 预扣后异常没有回补
5. 前端 Mock/API 模式搞混
6. 新增接口后没有在前端统一解包 Result<T>
7. 新增页面后忘记加路由
8. 新增表后没有更新 init.sql 或迁移 SQL
```
